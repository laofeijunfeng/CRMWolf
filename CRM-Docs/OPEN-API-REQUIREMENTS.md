# CRM 业务链路开放接口需求规格说明书

## 文档信息

| 项目 | 内容 |
|------|------|
| 文档版本 | v1.0 |
| 编制日期 | 2026-04-23 |
| 适用系统 | CRMWolf 系统 |
| 编制目的 | 明确CRM系统开放接口的开发需求、设计规范、业务规则，指导开放接口的研发、测试与交付 |

## 一、引言

### 1.1 背景

CRMWolf 系统已具备完整的CRM核心业务链路（线索→客户→商机→合同→回款→发票）及内部API体系，为支撑外部系统（如合作伙伴平台、业务中台、第三方系统）集成需求，需基于现有业务逻辑和内部API，设计并开发一套标准化、安全化的开放接口，实现核心业务能力的对外输出。

### 1.2 范围

本说明书覆盖CRMWolf系统核心业务模块的开放接口开发需求，包括：线索、客户、商机、合同、回款、发票六大模块；不包含系统内部管理类接口（如权限配置、流程模板管理）的开放。

### 1.3 术语与定义

| 术语 | 定义 |
|------|------|
| 开放接口 | 对外暴露的HTTP API，供外部系统调用，实现跨系统数据交互和业务操作 |
| API Key | 开放接口的系统级认证凭证，用于识别调用方身份 |
| 外部系统 | 调用本开放接口的第三方平台、业务系统等 |
| 核心链路 | 线索→客户→商机→合同→回款→发票的完整销售业务流程 |
| 枚举值 | 各业务状态的标准化编码（如线索状态NEW/CONTACTED等） |

## 二、总体需求

### 2.1 设计原则

1. **兼容性**：开放接口需兼容现有内部业务逻辑，数据结构、状态枚举与内部API保持一致；
2. **安全性**：所有开放接口需接入认证鉴权机制，防止未授权访问；
3. **易用性**：接口命名规范、参数清晰，提供完整的接口文档和调用示例；
4. **稳定性**：接口响应格式统一，错误码标准化，保障外部系统调用的稳定性；
5. **可扩展性**：接口设计预留扩展空间，支持后续新增业务场景的扩展。

### 2.2 接口架构

开放接口采用分层设计，与内部系统解耦，架构如下：

```
外部系统 → 开放接口层（认证/鉴权/参数校验） → 业务服务层（复用内部业务逻辑） → 数据访问层
```

### 2.3 通用规范

#### 2.3.1 通信协议

- 传输协议：HTTPS
- 请求方法：RESTful风格（POST/GET/PUT/PATCH/DELETE）
- 字符编码：UTF-8

#### 2.3.2 数据格式

**请求格式**：JSON

**响应格式**：统一返回结构，示例如下：

```json
{
  "code": 0,          // 错误码（0=成功，非0=失败）
  "message": "success", // 提示信息
  "data": {},         // 业务数据（成功时返回）
  "timestamp": 1704067200 // 响应时间戳（秒级）
}
```

#### 2.3.3 错误码规范

复用内部系统错误码体系，核心错误码如下：

| 错误码 | 说明 |
|--------|------|
| 0 | 成功 |
| 400 | 参数错误（如必填参数缺失、格式错误） |
| 401 | 未认证（API Key无效/缺失） |
| 403 | 无权限（调用方无该接口访问权限） |
| 404 | 资源不存在（如线索ID不存在） |
| 409 | 资源冲突（如重复创建线索、线索已转化） |
| 500 | 服务器内部错误 |

#### 2.3.4 分页参数

所有列表查询接口统一分页参数：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| page | Integer | 否 | 页码，默认1 |
| page_size | Integer | 否 | 每页条数，默认20，最大100 |

分页响应数据格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [],          // 数据列表
    "total": 100,        // 总条数
    "total_pages": 5,    // 总页数
    "page": 1,           // 当前页码
    "page_size": 20      // 当前页条数
  },
  "timestamp": 1704067200
}
```

#### 2.3.5 时间格式

- 日期参数：YYYY-MM-DD
- 时间参数：YYYY-MM-DD HH:mm:ss
- 时间戳：Unix秒级时间戳

## 三、各模块开放接口需求

### 3.1 线索模块开放接口（/open/v1/leads）

#### 3.1.1 接口列表

| 方法 | 接口路径 | 功能描述 | 调用方权限 |
|------|----------|----------|------------|
| POST | / | 外部系统录入线索 | 所有授权调用方 |
| GET | / | 查询线索列表（支持筛选） | 所有授权调用方 |
| GET | /{lead_id} | 获取线索详情 | 所有授权调用方 |
| GET | /{lead_id}/status | 查询线索状态 | 所有授权调用方 |
| POST | /{lead_id}/convert | 线索转化为客户（需校验权限） | 核心业务系统 |

#### 3.1.2 核心接口详情

##### （1）创建线索接口

- 接口路径：`POST /open/v1/leads`
- 功能描述：外部系统录入新线索，同步至CRMWolf系统公海池或指定负责人
- 请求参数：

```json
{
  "lead_name": "张三",        // 必填，线索名称（联系人姓名）
  "phone": "13800138000",    // 必填，联系电话
  "source": "WEBSITE",       // 必填，线索来源（WEBSITE/REFERRAL/EVENT/COLD_CALL）
  "company": "XX科技有限公司",// 选填，所属企业
  "industry": "互联网",      // 选填，行业
  "assign_user_id": 123      // 选填，指定负责人ID（不填则进入公海池）
}
```

- 响应数据：

```json
{
  "code": 0,
  "message": "创建成功",
  "data": {
    "lead_id": 1001,         // 线索ID
    "status": "NEW"          // 初始状态为新线索
  },
  "timestamp": 1704067200
}
```

- 业务规则：
  - 手机号需符合格式校验；
  - 线索来源需为枚举值范围内；
  - 创建成功后线索状态默认为NEW。

##### （2）线索转化接口

- 接口路径：`POST /open/v1/leads/{lead_id}/convert`
- 功能描述：外部系统触发线索转化，生成客户及联系人
- 请求参数：

```json
{
  "account_name": "XX科技有限公司",  // 必填，客户名称
  "industry": "互联网",              // 必填，客户行业
  "default_procurement_method_id": 1 // 必填，默认采购方式ID
}
```

- 响应数据：

```json
{
  "code": 0,
  "message": "转化成功",
  "data": {
    "customer_id": 2001,
    "contact_id": 3001,
    "lead_id": 1001
  },
  "timestamp": 1704067200
}
```

- 业务规则：
  - 校验线索存在且状态非CONVERTED；
  - 自动创建客户（关联source_lead_id）和主联系人；
  - 线索状态更新为CONVERTED。

### 3.2 客户模块开放接口（/open/v1/customers）

#### 3.2.1 接口列表

| 方法 | 接口路径 | 功能描述 | 调用方权限 |
|------|----------|----------|------------|
| GET | / | 查询客户列表（支持筛选） | 所有授权调用方 |
| GET | /{customer_id} | 获取客户详情 | 所有授权调用方 |
| GET | /{customer_id}/opportunities | 查询客户关联商机 | 所有授权调用方 |
| GET | /{customer_id}/contracts | 查询客户关联合同 | 所有授权调用方 |
| PATCH | /{customer_id}/status | 更新客户状态 | 核心业务系统 |

#### 3.2.2 核心接口详情

##### 查询客户详情接口

- 接口路径：`GET /open/v1/customers/{customer_id}`
- 功能描述：获取客户基本信息、联系人、状态等详情
- 请求参数：无（路径参数customer_id必填）
- 响应数据：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "customer_id": 2001,
    "account_name": "XX科技有限公司",
    "industry": "互联网",
    "status": 0,                // 客户状态（0=跟进中，1=已成交，2=已流失，3=非激活）
    "status_name": "跟进中",
    "primary_contact": {        // 主联系人信息
      "contact_id": 3001,
      "name": "张三",
      "phone": "13800138000"
    },
    "create_time": "2024-01-01 10:00:00"
  },
  "timestamp": 1704067200
}
```

- 业务规则：
  - 校验customer_id存在；
  - 状态编码与名称需同时返回，提升易用性。

### 3.3 商机模块开放接口（/open/v1/opportunities）

#### 3.3.1 接口列表

| 方法 | 接口路径 | 功能描述 | 调用方权限 |
|------|----------|----------|------------|
| POST | / | 外部创建商机 | 核心业务系统 |
| GET | / | 查询商机列表（支持筛选） | 所有授权调用方 |
| GET | /{opportunity_id} | 获取商机详情 | 所有授权调用方 |
| GET | /{opportunity_id}/status | 查询商机状态 | 所有授权调用方 |
| PATCH | /{opportunity_id}/win | 标记商机赢单 | 核心业务系统 |
| PATCH | /{opportunity_id}/lose | 标记商机输单 | 核心业务系统 |

#### 3.3.2 核心接口详情

##### 标记商机赢单接口

- 接口路径：`PATCH /open/v1/opportunities/{opportunity_id}/win`
- 功能描述：外部系统触发商机赢单，更新商机状态并同步客户状态（可选）
- 请求参数：

```json
{
  "actual_amount": 50000.00,    // 必填，实际成交金额
  "actual_closing_date": "2024-01-15", // 必填，成交日期
  "sync_customer_status": true  // 选填，是否同步更新客户状态为已成交（默认false）
}
```

- 响应数据：

```json
{
  "code": 0,
  "message": "赢单标记成功",
  "data": {
    "opportunity_id": 4001,
    "status": 1,                // 商机状态（1=赢单）
    "customer_status_updated": true // 客户状态是否更新
  },
  "timestamp": 1704067200
}
```

- 业务规则：
  - 校验商机存在且状态为0（跟进中）；
  - 记录实际成交金额和日期；
  - 若sync_customer_status为true，同步更新关联客户状态为1（已成交）。

### 3.5 回款模块开放接口（/open/v1/payments）

#### 3.5.1 接口列表

| 方法 | 接口路径 | 功能描述 | 调用方权限 |
|------|----------|----------|------------|
| POST | /contracts/{contract_id}/payment-plans | 创建回款计划 | 核心业务系统 |
| GET | /payment-plans | 查询回款计划列表 | 所有授权调用方 |
| POST | /payment-plans/{plan_id}/records | 登记回款 | 核心业务系统 |
| GET | /contracts/{contract_id}/payment-summary | 查询合同回款汇总 | 所有授权调用方 |
| GET | /reminders/overdue | 查询逾期回款提醒 | 所有授权调用方 |

#### 3.5.2 核心接口详情

##### 登记回款接口

- 接口路径：`POST /open/v1/payments/payment-plans/{plan_id}/records`
- 功能描述：外部系统登记实际回款，更新回款计划状态
- 请求参数：

```json
{
  "actual_amount": 20000.00,  // 必填，实际回款金额
  "payment_time": "2024-02-01", // 必填，回款时间
  "payment_method": "TRANSFER", // 选填，回款方式（转账/现金/支票）
  "remark": "首付回款"         // 选填，备注
}
```

- 响应数据：

```json
{
  "code": 0,
  "message": "回款登记成功",
  "data": {
    "record_id": 6001,
    "plan_id": 7001,
    "plan_status": "COMPLETED" // 回款计划状态（PENDING/OVERDUE/PARTIAL/COMPLETED）
  },
  "timestamp": 1704067200
}
```

- 业务规则：
  - 校验回款计划存在且合同状态为SIGNED/EFFECTIVE；
  - 累计回款金额达到计划金额时，计划状态更新为COMPLETED；
  - 累计回款金额未达计划金额时，计划状态更新为PARTIAL；
  - 回款时间晚于计划日期且未全额回款时，计划状态更新为OVERDUE。

## 四、安全需求

### 4.1 认证机制

- 所有开放接口需接入API Key认证，调用方需在请求头中携带`X-API-Key`参数；
- API Key需与调用方系统绑定，支持权限粒度控制（如部分调用方仅可查询，不可修改）；
- 支持API Key的启用/禁用、有效期管理，禁用后无法调用接口。

### 4.2 数据安全

- 敏感数据（如手机号、银行卡号）传输时需脱敏，响应数据中隐藏中间位；
- 接口调用日志需完整记录（调用方、接口路径、请求参数、响应结果、调用时间），日志留存不少于6个月；
- 防止重放攻击：支持请求时间戳（`X-Timestamp`）和签名（`X-Signature`）机制，签名失效时间默认5分钟。

### 4.3 权限控制

- 基于调用方维度配置接口访问权限，如：
  - 合作伙伴系统：仅可调用线索查询、客户查询接口；
  - 核心业务系统：可调用全量接口（含创建、修改、删除）；
  - 财务系统：仅可调用回款、发票相关接口。

## 五、非功能需求

### 5.1 性能需求

- 接口响应时间：查询类接口≤500ms，写入类接口≤1000ms；
- 接口并发能力：支持至少100 TPS的并发调用，无服务降级；
- 峰值处理：支持每秒200次调用的峰值，需做限流保护（返回429错误码）。

### 5.2 可靠性需求

- 接口可用性：≥99.9%（全年故障时间≤8.76小时）；
- 容错能力：接口调用失败时，需返回明确的错误码和提示信息，便于排查；
- 重试机制：写入类接口需支持幂等性设计，防止重复提交（如通过请求ID去重）。

### 5.3 兼容性需求

- 支持主流HTTP客户端调用（如Postman、curl、Java/PHP/Python SDK）；
- 响应数据格式需兼容JSON解析库，避免特殊字符、嵌套过深。

## 六、测试要求

### 6.1 功能测试

- 覆盖所有接口的正常场景、异常场景（如参数缺失、资源不存在、权限不足）；
- 验证业务规则的正确性（如线索转化后状态更新、回款计划金额校验）；
- 验证接口幂等性（重复调用写入接口不产生重复数据）。

### 6.2 性能测试

- 并发测试：模拟100 TPS并发调用，验证接口响应时间和系统稳定性；
- 压力测试：逐步提升并发量至200 TPS，验证系统限流机制是否生效。

### 6.3 安全测试

- 认证测试：验证无API Key、无效API Key时接口是否拒绝访问；
- 权限测试：验证低权限调用方无法调用高权限接口；
- 数据脱敏测试：验证敏感数据传输和响应时是否脱敏。
