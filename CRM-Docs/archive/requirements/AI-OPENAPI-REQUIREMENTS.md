# CRMWolf AI 专用 OpenAPI 需求文档

> **状态：✅ 已实现** | 完成日期：2026-05-25
> 版本：2.0 | 创建日期：2026-05-25

---

## 一、 背景与核心痛点

### 1.1 现状
CRMWolf 采用标准 RESTful API 架构（`/api/v1/`），面向**人类用户通过浏览器界面操作**设计。后端基于 FastAPI + SQLAlchemy，业务逻辑集中在 `services/` 层，数据模型为典型的实体关系型（客户/商机/跟进独立成表）。

### 1.2 核心痛点
当前系统对接 AI 时，面临典型的**"三层架构不匹配"**问题：

| 层级 | 现有设计 | AI 使用方式 | 不匹配表现 |
|------|---------|------------|-----------|
| **数据层** | 实体中心（独立表结构） | 事件中心（一次交互包含多实体变更） | AI 需多次跨表调用才能完成一个业务闭环 |
| **接口层** | 严格 CRUD（必填字段多、强校验） | 模糊自然语言意图 | AI 无法一次性补全 10+ 必填字段，易触发校验拦截 |
| **逻辑层** | 规则硬编码在代码中 | 无法读取隐性业务规则 | AI 不知晓"签合同"→阶段自动切"合同谈判"→赢率变 80% 等联动逻辑 |

### 1.3 典型场景举例
**用户输入**：`"今天跟李四聊了，他们想做个CRM，预算30万，下个月启动"`

| 现有 API 调用链 | AI 面临的障碍 |
|----------------|-------------|
| 1. `GET /customers/?name=李四` 查找客户 | 如何从自然语言准确提取"李四"并匹配 ID？ |
| 2. `POST /follow-ups/` 创建跟进 | 跟进记录应关联客户还是商机？字段如何映射？ |
| 3. `POST /opportunities/` 创建商机 | 需填 10+ 字段，AI 仅提取部分信息，如何初始化？ |
| 4. `POST /opportunities/{id}/advance` 推进阶段 | AI 不知道"需求确认"阶段默认对应 20% 赢率 |
| 5. 关联跟进与商机 | 现有 API 缺乏直接的跨实体关联接口 |

**结论**：现有 API 并非为 AI 设计。强行对接会导致大量错误处理逻辑、不可维护的 Prompt 堆砌，以及频繁的人工纠错。

## 二、 需求目标与设计原则

### 2.1 核心目标
创建一套专门供 AI 调用的 OpenAPI 接口层（路径前缀 `/ai/`），作为自然语言意图与现有 CRM 系统之间的**翻译层**与**编排层**。

### 2.2 设计原则
| 原则 | 说明 |
|------|------|
| **原子化** | 接口粒度最小化，每个接口只做一件事，无副作用，支持独立回滚 |
| **预览优先** | 所有写操作必须支持 `preview=true`，先返回变更计划，确认后再执行 |
| **幂等性** | 基于 `action_id` 保证操作唯一性，重复调用不产生副作用 |
| **无状态** | OpenAPI 本身保持无状态，会话状态与多轮对话逻辑由应用层维护 |
| **向后兼容** | 现有 `/api/v1/` 完全保持不变，`/ai/` 为纯新增层，不侵入现有代码 |

### 3.1 整体分层架构
```
┌─────────────────────────────────────────────────────────┐
│              AI 客户端 (MCP / CLI / 飞书 / 企微)          │
└──────────────────────────┬──────────────────────────────┘
                           │ 自然语言意图
                           ▼
┌─────────────────────────────────────────────────────────┐
│              第一层：业务意图层 (Business Intent)          │
│  - 意图解析 (Parse)  - 实体提取 (Extract) - 规则匹配     │
└──────────────────────────┬──────────────────────────────┘
                           │ 结构化动作序列
                           ▼
┌─────────────────────────────────────────────────────────┐
│            第二层：原子动作层 (Atomic Actions)             │
│  - 创建跟进 / 初始化商机 / 更新阶段 / 关联实体 / 编排执行   │
└──────────────────────────┬──────────────────────────────┘
                           │ 精确参数调用
                           ▼
┌─────────────────────────────────────────────────────────┐
│           第三层：系统元数据层 (System Metadata)           │
│  - 阶段赢率 / 业务规则库 / 实体关系 / 权限矩阵            │
└──────────────────────────┬──────────────────────────────┘
                           │ 路由至现有服务
                           ▼
┌─────────────────────────────────────────────────────────┐
│              现有 CRMWolf 核心层 (/api/v1/ + services/)  │
└─────────────────────────────────────────────────────────┘
```

## 四、 API 详细设计

### 4.1 第一层：业务意图层 (Intent Layer)
负责将模糊的自然语言转换为结构化数据。

| 接口 | 方法 | 核心职责 | 关键输入/输出 |
|------|------|---------|-------------|
| `/ai/intents/parse` | POST | **意图分类** | 输入：用户文本+上下文<br>输出：意图列表+置信度+推理依据 |
| `/ai/intents/extract` | POST | **实体抽取** | 输入：用户文本+目标实体类型<br>输出：结构化 JSON 实体+缺失字段提示 |
| `/ai/rules/match` | GET | **规则匹配** | 输入：意图ID+实体数据<br>输出：匹配的业务规则+推荐动作序列 |

**设计要点**：
- 意图分类采用固定枚举池，避免 LLM 幻觉输出非标意图。
- 实体提取需支持上下文继承（如用户说"把那个改一下"，需结合历史对话补全实体）。
- 规则匹配支持向量检索或关键词匹配，返回 Top-N 最匹配规则。

### 4.2 第二层：原子动作层 (Action Layer)
负责执行具体的系统操作，所有接口必须支持 **Preview（预览）** 模式。

| 接口 | 核心职责 | 关键参数 |
|------|---------|---------|
| `/ai/actions/create-follow-up` | 创建跟进记录 | `customer_id`, `content`, `follow_up_time` |
| `/ai/actions/init-opportunity` | 初始化商机(最小集) | `customer_id`, `opportunity_name` |
| `/ai/actions/update-amount` | 更新商机金额 | `opportunity_id`, `amount` |
| `/ai/actions/update-stage` | 更新商机阶段(自动带赢率) | `opportunity_id`, `stage` |
| `/ai/actions/link-opportunity` | 关联商机与跟进记录 | `opportunity_id`, `follow_up_id` |
| `/ai/actions/set-reminder` | 设置跟进提醒 | `customer_id`, `reminder_time`, `content` |
| `/ai/actions/orchestrate` | **编排执行(多动作组合)** | `actions[]`, `depends_on`, `confidence_threshold` |

**通用交互协议**：
```json
// 请求体通用结构
{
  "preview": true,               // true=仅预览，false=执行
  "action_id": "act_xxx",        // 幂等ID（执行时必填）
  "params": { ... }              // 动作特定参数
}

// 响应体通用结构
{
  "action_id": "act_xxx",
  "status": "preview|completed|failed",
  "plan": {                      // preview 时返回
    "description": "将商机A金额从30万改为35万",
    "changes": [
      { "field": "amount", "from": 300000, "to": 350000 }
    ]
  },
  "confidence": 0.92,
  "requires_confirmation": true  // 是否需人工确认
}
```

### 4.3 第三层：系统元数据层 (Metadata Layer)
负责向 AI 暴露系统静态配置与业务上下文，使 AI "读懂"系统规则。

| 接口 | 核心职责 | 返回内容示例 |
|------|---------|-------------|
| `/ai/metadata/stages` | 商机阶段与赢率映射 | `[{"name":"需求确认","win_rate":0.2,"next_stages":["方案报价"]}]` |
| `/ai/metadata/rules` | 业务规则库 | `[{"trigger":"keyword","keyword":"签合同","action":"update_stage","target":"商务谈判"}]` |
| `/ai/metadata/entities` | 实体关系与必填字段 | `[{"name":"Opportunity","required":["customer_id","name"],"relations":["Customer"]}]` |
| `/ai/metadata/permissions`| 当前用户权限范围 | `{"data_scope":"own","restricted_actions":["delete_contract"]}` |

## 五、 多轮对话与交互机制

### 5.1 职责边界划分
| 层级 | 被责 | 不做什么 |
|------|------|---------|
| **OpenAPI 层** | 提供无状态的原子能力（预览、执行、查元数据） | 不维护会话状态、不处理等待逻辑、不管理对话流 |
| **应用层** | 维护对话上下文、状态机流转、用户交互 UI | 不直接操作数据库、不硬编码业务规则 |

### 5.2 多轮对话标准流程
1. **意图识别**：应用层调用 `/ai/intents/parse` 获取意图。
2. **置信度判断**：
   - `≥ 阈值`：直接进入执行准备。
   - `< 阈值`：应用层生成追问话术，暂停流程，等待用户补充。
3. **预览确认**：调用 `/ai/actions/*` (preview=true)，获取 `changes` 详情。
4. **人机交互**：应用层将 `changes` 渲染为确认卡片推送给用户。
5. **执行/修正**：
   - 用户确认：应用层携带 `action_id` 调用执行接口。
   - 用户修改：应用层重新组装参数，再次请求预览。
6. **结果反馈**：返回执行结果，更新对话上下文。

**示例交互**：
> **用户**：`"把那个30万的商机改成35万"`
> **应用层**：调用提取接口，发现匹配到 2 个商机 → 置信度低 → 暂停
> **机器人**：`"找到两个相近商机，您指的是哪个？\n1. 李四CRM项目 (30万)\n2. 王五OA项目 (30.5万)"`
> **用户**：`"李四那个"`
> **应用层**：调用预览接口 → 获取变更详情 → 推送确认卡片
> **机器人**：`"即将把【李四CRM项目】金额 30万→35万，确认执行吗？"`
> **用户**：`"确认"`
> **应用层**：携带 action_id 执行 → 返回成功结果

## 六、 安全、权限与审计设计

### 6.1 认证与鉴权
- **统一认证**：复用现有 JWT Bearer Token 体系，不新增认证协议。
- **双重鉴权**：MCP Server 等外部调用需额外校验 API Key，防止接口滥用。

### 6.2 风险分级控制
| 风险等级 | 动作类型 | 执行策略 |
|---------|---------|---------|
| **低风险** | 创建跟进、设置提醒 | 置信度 ≥ 0.85 自动执行 |
| **中风险** | 更新阶段/金额、关联实体 | 置信度 ≥ 0.90 自动执行，否则触发确认 |
| **高风险** | 赢单/输单、线索转化、删除 | **必须**人工确认，禁止自动执行 |
| **极高风险** | 审批流、财务操作 | AI 仅生成待办/草稿，**禁止**代执行 |

### 6.3 操作审计
- 所有通过 `/ai/` 接口执行的操作，底层日志强制标记 `source="ai"`。
- 记录完整链路：`intent_id` → `action_id` → `execution_id`，支持全链路追溯。
- 提供 `GET /ai/logs` 接口供应用层查询 AI 操作历史，支持按时间/用户/实体过滤。

## 七、 生态集成需求

OpenAPI 设计需预留标准扩展点，支持后续快速封装为不同形态的客户端：

| 集成形态 | 需求说明 | 技术路径 |
|---------|---------|---------|
| **MCP Server** | 供 Cursor/Claude 等 AI 编程助手或 Agent 框架调用 | 基于 OpenAPI Spec 自动生成 MCP Tools 定义 |
| **CLI 工具** | 供开发调试、批量数据修复、离线操作使用 | 封装 HTTP 客户端，支持 JSON 输入输出 |
| **IM 机器人** | 供飞书/企微/钉钉等即时通讯工具对接，实现无界面操作 | 接收 IM 消息 → 调用 `/ai/` → 渲染富文本卡片回传 |

## 八、 与现有开放接口的关系

### 8.1 两套接口对比

| 维度 | `/open/v1/` (v1.0) | `/ai/` (v2.0) |
|------|---------------------|---------------|
| **目标用户** | 外部系统（合作伙伴、业务中台） | AI Agent（MCP、CLI、IM机器人） |
| **调用方式** | 程序化调用，参数精确 | 自然语言意图驱动 |
| **字段要求** | 严格必填校验 | 最小化字段集，支持渐进补全 |
| **执行模式** | 直接执行 | Preview → 确认 → 执行 |
| **业务规则** | 调用方需自行理解 | 通过元数据接口暴露 |
| **错误处理** | 返回错误码 | 返回缺失字段提示 + 推荐动作 |

### 8.2 共享基础设施
两套接口共享以下基础设施：
- 认证体系（JWT + API Key）
- 权限校验逻辑
- 核心业务服务层（services/）
- 数据模型层
