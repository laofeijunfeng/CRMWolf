# 开发决策框架

> 每次开发前使用此框架评估方案，选择成熟、稳定的解决方案。

---

## 开发前三问

### Q1: 行业标准方案是什么？

**参考来源**：
- Spring Boot / Django / FastAPI 官方最佳实践
- Google / Meta / 阿里巴巴 技术规范
- 成熟开源项目的做法（如 Vue 官方文档、Element Plus）

**示例**：
| 场景 | 行业标准 |
|------|----------|
| API 请求 | axios + baseURL 统一配置 |
| 状态管理 | Pinia/Vuex store，而非组件间共享 ref |
| 数据库操作 | ORM + CRUD 层，而非直接 query |
| 环境一致性 | 代理转发，而非直连后端 |

---

### Q2: 本项目现有最佳实践是什么？

**查找方式**：
- 查看 [README.md](README.md) 按场景或关键词查找
- 复用已有配置/组件/模式
- 不重复造轮子

**示例**：
| 场景 | 本项目方案 |
|------|----------|
| API 请求 | `request.get('/v1/xxx')` 自动拼接 baseURL |
| SSE 请求 | `/api/v1/xxx` 完整路径 |
| 团队隔离 | CRUD 层统一传入 team_id |

---

### Q3: 简单方案 vs 成熟方案成本对比

**成本评估模型**：

| 成本类型 | 简单方案 | 成熟方案 |
|----------|----------|----------|
| **实现成本** | 低（1小时） | 中（2小时） |
| **排查成本** | 高（4小时 × N次） | 无 |
| **维护成本** | 高（环境差异、理解难度） | 低（统一逻辑） |
| **总成本** | 1 + 4N + 高维护 | 2 + 0 + 低维护 |

**决策原则**：

```
当 N ≥ 1 时（至少排查一次问题），成熟方案更优

排查成本 > 实现成本 → 选择成熟方案
```

## 成本对比实例

### 案例：SSE 请求路径

| 方案 | 实现成本 | 排查成本 | 后果 |
|------|----------|----------|------|
| `${baseURL}/v1/xxx`（简单） | 10 分钟 | 4 小时 × 1 次 | 本地正常，生产 405 |
| `/api/v1/xxx`（成熟） | 20 分钟 | 0 | 环境一致，无问题 |

**结论**：多花 10 分钟，节省 4 小时排查时间。

---

### 案例：team_id 传递

| 方案 | 实现成本 | 排查成本 | 后果 |
|------|----------|----------|------|
| 直接 query（简单） | 5 分钟 | 4+ 小时 | DATABASE_ERROR，定位难 |
| CRUD 层传 team_id（成熟） | 15 分钟 | 0 | 约束自动校验 |

## 强制检查项

每次开发必须检查以下项目：

### 环境一致性
- [ ] 本地开发路径和生产路径是否一致？
- [ ] 是否依赖某个环境变量才能正常工作？
- [ ] SSE/WebSocket/文件上传等特殊请求是否复用了统一配置？

### 配置复用
- [ ] 新代码是否复用了现有配置（如 baseURL）？
- [ ] 是否有硬编码路径/URL？
- [ ] 是否有只在某环境才能工作的代码？

### 团队隔离
- [ ] CRUD 操作是否传入 team_id？
- [ ] 是否绕过 CRUD 层直接操作数据库？
- [ ] Model 层约束是否被正确处理？

### 代码质量
- [ ] 是否复用了已有组件/工具函数？
- [ ] 是否遵循现有命名/结构规范？
- [ ] 是否有分散的重复逻辑？

## 反模式清单

以下做法被证明会导致问题，**禁止使用**：

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 硬编码完整 URL | 环境不一致 | 使用 baseURL 或相对路径 |
| `${VITE_API_BASE_URL}/v1/xxx` | 变量空时路径错误 | `/api/v1/xxx` 固定格式 |
| 直连后端 `localhost:8000` | 生产环境失效 | 通过代理转发 |
| 绕过 CRUD 层 | team_id 缺失 → 数据库错误 | CRUD 层统一入口 |
| 组件间共享 ref | 状态混乱、难以追踪 | Pinia store |
| 分散的表单校验 | 逻辑重复、难以维护 | Schema 校验 |
