# 结果沉淀导航

**Changelog 目录**：记录功能实施、缺陷修复、技术重构的结果沉淀

---

## 目录结构

```
changelog/
├── enhancements/              ← 功能优化（按需求汇总）
│   └── YYYY-MM-DD-feature-name.md
│
├── issues/                    ← 缺陷修复（重大缺陷单独）
│   └── YYYY-MM-DD-bug-fix.md
│
├── technical/                 ← 技术重构
│   └── YYYY-MM-DD-refactor-name.md
│
└── README.md                  ← 本导航文档
```

## 创建规则

### 创建时机

| 场景 | 创建要求 | 创建者 |
|------|----------|--------|
| 功能上线 | 必须创建 changelog | 开发者 |
| 重大缺陷修复 | 必须创建 changelog | 开发者 |
| 复杂技术重构 | 必须创建 changelog | 开发者 |
| 小修复/优化 | 可选 | 开发者 |

### 文档命名

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| 功能优化 | `YYYY-MM-DD-[主题].md` | `2026-06-12-ai-openapi.md` |
| 缺陷修复 | `YYYY-MM-DD-bug-[主题].md` | `2026-06-12-sse-parser-fix.md` |
| 技术重构 | `YYYY-MM-DD-refactor-[主题].md` | `2026-06-12-team-isolation.md` |

## Changelog 与归档关系

```
需求/计划完成 (status: completed)
    ↓
创建 Changelog 文档（开发者）
    ↓
CI 检测 Changelog 存在
    ↓
触发自动归档 → archive/
```

**前置条件**：归档前必须有对应的 Changelog 文档

|----------|----------|

| 了解重构的架构变更 | `technical/YYYY-MM-DD-refactor-xxx.md` |
| 查看历史需求归档 | `../archive/README.md` |

## 最佳实践

### 实施总结内容

| 章节 | 内容要求 |
|------|----------|
| 实施结果 | 功能完成情况表格 |
| 技术实现要点 | 核心改动 + 技术难点 |
| 遗留问题 | 未完成项 + 处理建议 |
| 经验沉淀 | 最佳实践 + 教训 |

### 关联文档

每个 Changelog 必须包含：

```markdown
**关联需求**：[需求文档链接]
**关联计划**：[计划文档链接]
**关联 PR**：#XXX
```

## 禁止行为

| 禁止 | 原因 |
|------|------|
| 未创建 Changelog 即归档 | 违反结果沉淀规则 |
| Changelog 缺少关键章节 | 信息不完整 |
| Changelog 未关联需求/计划 | 无法追溯源头 |
