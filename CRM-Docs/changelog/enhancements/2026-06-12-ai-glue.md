## 二、技术实现要点

### 核心架构

**状态机流程**：
```
IDLE → COLLECTING → RESOLVING_ENTITY → RESOLVING_AMBIGUITY → PREVIEW → EXECUTING
```

**LLM 增强组件**：
- IntentDetector：意图识别
- EntityResolver：实体消解
- CorrectionResolver：修正解析
- ConfirmationDetector：确认检测
- CancellationDetector：取消检测
- DisambiguationResolver：歧义消解

**多意图解析**：
- 复合指令分解
- 依赖关系处理
- 顺序执行编排

### 技术难点

1. **实体消解精度**
   - 问题：自然语言中的模糊引用难以精确匹配
   - 解决方案：LLM 增强实体解析 + 三阶降级策略
   - 经验：结合数据库查询 + 向量搜索，精度提升 80%

2. **歧义消解效率**
   - 问题：多实体歧义时用户等待时间长
   - 解决方案：预加载候选列表 + 快速展示 UI
   - 经验：提前查询减少延迟，用户体验提升

3. **Preview 模式强制**
   - 问题：Preview 机制复杂，易遗漏
   - 解决方案：统一入口 + 强制 Preview-before-Write
   - 经验：ActionEntry 统一入口，避免违规调用

## 四、经验沉淀

### 最佳实践

- **单一写入者原则**：所有数据库写操作通过 ActionExecutor
- **Preview 强制规则**：Preview-before-Write，无 Preview 不执行
- **LLM 增强策略**：关键词匹配 + LLM 补充，成本可控
- **幂等性保障**：action_id + Redis Session，防止重复执行

### 教训

- **双份 Preview 违规**：早期 ActionExecutor 和 Handler 都生成 Preview，违反单一来源
- **路由违规调用**：早期 ActionExecutor 直接调用 CRUD，违反分层边界
- **修复**：统一入口 + 路由对齐整改（见 AI-GLUE-ROUTING-ALIGNMENT-RFC）
