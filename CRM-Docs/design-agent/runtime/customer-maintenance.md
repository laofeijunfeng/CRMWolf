# 客户资料维护

- **用途：**定义 Agent 维护客户基础信息的能力边界。
- **适用范围：**联系人、发票抬头、部署信息、客户成员和 License 前置资料。
- **权威性：**本文件约束客户资料维护类 tool 的交互和执行。
- **相关规范：**[Tool 规范](tools.md) · [交互编排策略](interaction-policy.md) · [Memory 规范](memory.md)

## 定位

客户资料维护是跟进记录之外的高频轻量场景。

资料维护类动作可以不先创建跟进记录，但必须先识别客户并确认写入。

所有写入必须调用现有 CRM API，不能直接改业务数据表。

## 已接入能力

创建联系人：

- 识别客户、姓名、手机号、职务、性别、邮箱、是否决策人。
- 必填字段不足时，只追问缺失字段。
- 字段完整时，先确认，再调用 `POST /v1/customers/{customer_id}/contacts`。

创建发票抬头：

- 识别抬头类型、抬头名称、税号、开户行、账号、地址、电话、是否默认。
- 必填字段不足时，只追问缺失字段。
- 字段完整时，先确认，再调用现有发票抬头 API。

创建部署信息：

- 识别部署名称、服务器地址、授权人数、是否默认。
- 必填字段不足时，只追问缺失字段。
- 字段完整时，先确认，再调用现有部署信息 API。

设置客户成员：

- 必须先读取客户成员候选人 API。
- 用户说出成员姓名时，由 LLM 解析意图，代码层匹配候选人。
- 候选人不唯一时先让用户选择。
- 写入时调用现有客户成员 API。

## 待接入能力

License 申请：

- 试用 License 依赖审批通过的商机和部署信息。
- 正式 License 依赖审批通过的合同和部署信息。
- 用户未说明试用或正式时，先追问 License 类型。
- 第一版只在前置条件明确、字段完整并经用户确认后调用现有 API。

## 交互规则

同一轮只输出一个动作。

如果既缺字段又需要确认，先补字段，再确认。

如果客户不唯一，先选客户，再进入字段补充或确认。

如果用户在补资料时明确开启新客户流程，按 pending 中断策略切换或追问。
