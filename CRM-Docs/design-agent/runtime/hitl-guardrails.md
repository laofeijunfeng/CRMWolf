# HITL 与 Guardrails

- **用途：**定义用户确认和执行保护机制。
- **适用范围：**所有写入类 tool、任务确认、自动化建议。
- **权威性：**本文件拥有 HITL 与 guardrails 规则。
- **相关规范：**[Tool 规范](tools.md) · [结构化输出](structured-output.md) · [交互编排策略](interaction-policy.md)

## 写入确认

创建跟进记录、联系人、发票抬头、部署信息、回款计划、回款登记等写入动作必须经过用户确认。

确认前 Agent 只能说明将要执行什么，不得声称已完成。

一轮回复只能有一个主确认目标。跟进记录需要确认时，不得在同一条最终回复里同时展示下一步建议。

跟进记录创建成功后，Agent 才能继续提出创建商机、推进阶段等下一步建议或第二个确认任务。

确认、补字段、选择对象和建议决策的先后顺序以交互编排策略为准。

## 执行保护

执行前必须校验：

- 当前 task 是否存在且未过期。
- 用户是否 approve。
- tool 是否在当前任务允许列表。
- 客户或业务对象是否在当前确认上下文内。
- payload 是否通过 Pydantic 校验。

## 建议保护

AI 结构化输出的建议不能直接展示为可执行下一步动作，必须先经过业务 guardrail。

建议动作必须来自当前支持的 action/tool 白名单；没有 tool 的能力不得展示给用户。

系统没有跟进提醒 tool；跟进时间只能作为跟进记录字段保存，不得建议“设置跟进提醒”。

回款链路必须校验：

- CREATE_PAYMENT_PLAN 必须绑定客户上下文中已有合同 ID。
- CREATE_PAYMENT_RECORD 必须绑定客户上下文中已有未完成回款计划 ID。
- 没有合同不得建议创建回款计划。
- 没有回款计划不得建议登记回款。

商机阶段推进必须校验：

- MOVE_OPPORTUNITY_STAGE 必须绑定客户上下文中的商机 ID。
- 商机必须是进行中且审批通过。
- 目标 `stage_template_id` 必须来自该商机动态采购阶段列表。
- 目标阶段不能是当前阶段；默认不得倒退到更早阶段。
- 真实推进必须等用户确认后调用现有 `move-stage` API。
- 同一轮还需创建跟进记录时，阶段推进必须作为跟进记录成功后的第二个确认任务。

License 建议必须校验：

- CREATE_LICENSE_APPLICATION 必须先有部署信息。
- 试用 License 必须绑定审批通过商机。
- 正式 License 必须绑定审批通过且可用合同。
- License 类型不明确时不得进入可执行建议。

## Pending 中断保护

挂起任务中断判断只负责路由，不代表用户批准任何写入。

自动切换新流程必须满足：

- AI structured output 判定为 `START_NEW_FLOW`。
- 置信度不低于 0.85。
- 本轮存在明确新客户或明确新业务意图。
- 本轮不像当前 pending 的字段补充。

不满足以上条件时，应继续 pending 或询问用户确认。

切换后新流程中的写入动作仍必须重新走 HITL。

## Edit/Reject

用户修改字段时，应更新待确认 payload，再次要求确认。

用户拒绝时，应关闭任务并保留审计记录。

## LangChain HITL

当引入受控 tool-calling 子 Agent 时，应接入 LangChain HITL middleware。

现有自定义 HITL 不应被移除，除非新链路能覆盖确认、审计、恢复和权限上下文。
