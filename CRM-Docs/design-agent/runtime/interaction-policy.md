# 交互编排策略

- **用途：**定义 Agent 与用户交互时的单轮动作原则。
- **适用范围：**确认、补字段、选择、建议、中断确认和普通回复。
- **权威性：**本文件拥有最终用户交互编排规则。
- **相关规范：**[HITL 与 Guardrails](hitl-guardrails.md) · [结构化输出](structured-output.md) · [Memory 规范](memory.md)

## 核心原则

同一轮对用户的最终回复，只能包含 1 个需要用户响应的交互动作。

交互动作包括：

- 语义澄清。
- 新流程中断确认。
- 客户或业务对象选择。
- 必填字段补充。
- 写入确认。
- 下一步建议决策。
- 普通查询结果后的继续操作询问。

不得在一条最终回复里同时要求用户确认、补字段、选择对象或决定建议。

## 优先级

最终回复前必须按以下优先级选择唯一交互动作：

1. 语义澄清。
2. 新流程中断确认。
3. 客户或业务对象选择。
4. 必填字段补充。
5. 写入确认。
6. 写入结果反馈。
7. 下一步建议决策。
8. 普通总结或查询结果。

低优先级动作只能进入 next task、内部建议事件或下一轮，不得抢占当前轮。

## 顺序规则

客户不唯一时，先选择客户，再确认写入。

必填字段不完整时，先补字段，再确认写入。

跟进记录是当前主产物时，先确认并创建跟进记录，再提出创建商机或推进阶段建议。

商机阶段推进、商机创建等二级动作必须在跟进记录成功后进入第二个确认或补字段任务。

跟进质量低于 60 分时，先补跟进质量；达到 60 分时，不再向用户索要质量补充。

写入结果反馈可以附带一个简短下一步问题，但该问题必须是唯一交互动作。

## LangChain 映射

LLM 只负责用 structured output 产出候选意图、字段、建议和置信度。

代码层使用 LangGraph 状态机选择当前唯一交互动作。

LangChain middleware 或等价 runtime hooks 负责最终回复前的交互 guardrail。

HITL 只暂停当前待执行动作，不代表允许同时收集多个用户决策。

Memory 必须保存 pending task、next task、已补字段和被延后的建议。

## 输出约束

最终回复必须短、明确，只问一个问题。

如果存在多个候选下一步，先问是否继续处理最高优先级动作。

如果用户回复“是”，只能推进当前 pending task，不得重新猜测旧建议。

如果用户开启新客户或新业务流程，必须按 pending 中断策略决定切换、继续或追问。

## 实现要求

新增或修改 Agent 响应逻辑时，必须经过 interaction planner。

planner 输入包括 semantic result、business context、suggestions、pending task 和 tool capability。

planner 输出只能有一个 `interaction_action`。

最终回复前必须校验没有多个用户响应目标。

违反本规则时，优先保留高优先级动作，其他动作写入 next task 或内部 trace。
