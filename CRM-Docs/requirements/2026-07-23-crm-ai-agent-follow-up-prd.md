# 围绕客户跟进记录的 CRM AI Agent 需求文档

## 1. 产品定位

本系统定位为：围绕客户跟进记录的智能客户关系管理系统。

CRM AI Agent 的核心目标不是做一个泛化聊天机器人，而是帮助销售把客户沟通、业务进展、下一步动作沉淀为高质量跟进记录，并基于跟进记录驱动商机、回款、发票、License、客户资料等业务动作。

Agent 必须始终基于系统现有业务 API 工作，绝对不允许直接操作数据库，绝对不允许绕过权限体系，绝对不允许绕过后端已有业务规则、审批规则和数据校验规则。

## 2. 核心目标

1. 降低销售录入客户跟进记录的成本

   销售可以用自然语言输入客户沟通情况，Agent 自动识别客户、跟进内容、下一步动作和下次跟进时间，在客户唯一明确时自动创建跟进记录。

2. 提升跟进记录质量

   跟进记录先创建，不阻塞销售录入；AI 异步评估跟进质量，在内容缺失、下一步不清晰、客户反馈不充分时提醒销售补充优化。

3. 基于跟进记录推动业务动作

   Agent 根据客户全量上下文和用户输入，主动识别后续动作，例如创建商机、推进商机、创建回款计划、登记回款、创建发票申请、创建 License 申请等。

4. 统一对话入口

   第一版采用统一 Agent 对话入口。销售可以直接输入自然语言，由 Agent 判断是查询、跟进录入、业务推进，还是客户资料维护。

## 3. 基本原则

1. 跟进记录是业务推进的核心上下文

   创建商机、推进商机、赢单/输单、创建回款计划、登记回款、创建发票申请、创建 License 申请等业务推进动作，必须先沉淀客户跟进记录。

2. 客户资料维护类动作可以不创建跟进记录

   创建联系人、创建发票抬头、创建部署信息、设置客户成员等资料维护动作，可以直接按明确命令执行，不强制补跟进记录。

3. 跟进记录可自动创建

   客户唯一明确时，Agent 可自动创建跟进记录。其他非跟进记录写入动作，按各自规则确认。

4. 所有 tool 必须封装系统现有 API

   Agent tool 只能调用后端 API，不允许直接查库、写库或绕过后端权限。用户当前权限不足时，Agent 必须反馈 API 权限错误，不得替用户越权执行。

5. 业务对象状态必须实时校验

   Agent 执行业务动作前，应通过系统 API 获取最新客户、合同、商机、回款计划、发票、License、成员等上下文，不能依赖过期缓存直接写入。

## 4. 用户输入类型

Agent 第一版需要识别以下输入类型：

1. 客户跟进类

   例如：今天和越秀金融王总沟通了项目进展，客户还在立项评估，下周三再确认。

2. 回款类

   例如：光大证券今天回款了。

3. 商机推进类

   例如：越秀金融这个项目进入采购评审了。这个项目赢单了，成交金额 30 万。

4. 发票类

   例如：光大证券这笔回款帮我申请发票。

5. License 类

   例如：给越秀金融申请一个试用 License，到期时间下月底。

6. 客户资料维护类

   例如：帮我给越秀金融创建联系人王总，电话 138...

   例如：给光大证券创建一个发票抬头。

   例如：把越秀金融协作成员设置为李四。

7. 查询类

   例如：光大证券现在有哪些合同和回款计划？

   查询类不创建跟进记录。

## 5. 客户识别规则

1. 客户唯一匹配时，Agent 可以继续处理。
2. 客户匹配不唯一时，Agent 不得自动创建跟进记录或执行业务动作，必须让用户选择客户。
3. 客户不存在时，第一版不由 Agent 创建客户；提示用户先在系统中创建客户。
4. 联系人不存在时，不阻塞跟进记录创建；联系人称呼保留在跟进内容中。Agent 可建议补充联系人，但不自动创建，除非用户明确发起创建联系人动作。

## 6. 跟进记录需求

跟进记录创建条件：

1. 用户输入包含客户沟通、业务进展、回款、开票、合同推进、License 推进等业务事实。
2. 客户唯一明确。
3. 跟进内容可从用户输入中形成。

跟进记录字段识别：

- 客户名称
- 跟进内容
- 跟进方式
- 下一步动作
- 下次跟进时间
- 涉及联系人称呼
- 涉及业务对象线索，如商机、合同、回款、发票、License

跟进方式规则：

- 用户明确说明电话、微信、拜访等方式时，按语义识别。
- 用户未说明时，默认使用“其他/未指定沟通”，不阻塞录入。

下次跟进规则：

- 识别到“下周三再确认”“月底前再跟进”等表达时，写入 `next_follow_time` 和 `next_action`。
- 第一版不额外创建待办或提醒。

质量评估规则：

- 跟进记录创建后，AI 异步评估质量。
- 质量评估不阻塞录入。
- 质量差时提醒销售补充，例如客户反馈不清楚、下一步动作不明确、缺少关键联系人、缺少业务阶段信息等。

## 7. Agent 主动建议

Agent 创建跟进记录后，应查询客户全量业务上下文，并主动判断是否存在下一步动作。

建议范围采用积极型策略：

- 不只基于用户原话，还要结合客户、商机、合同、回款计划、回款记录、发票、License、部署信息等上下文。
- 每轮最多展示 3 个高优先级建议。
- 建议排序依据：业务紧急度、确定性、与当前输入相关性。

用户拒绝建议时：

- 只记录在本次 Agent 会话上下文中。
- 不写入业务数据。
- 本轮不重复相同建议。

## 8. 第一版支持的写入动作

业务推进类动作，必须先有客户跟进记录上下文：

- 创建商机
- 推进商机阶段
- 标记商机赢单
- 标记商机输单
- 创建回款计划
- 登记回款
- 创建发票申请
- 创建 License 申请
- License 申请创建后询问是否提交审批

客户资料维护类动作，可以不创建跟进记录：

- 创建联系人
- 创建发票抬头
- 创建部署信息
- 设置客户成员

第一版不支持：

- 创建客户
- 创建合同
- 修改联系人
- 修改发票抬头
- 修改部署信息
- 修改回款计划
- 删除业务对象
- 撤销 Agent 操作
- 上传合同附件
- 上传发票文件
- Agent 直接审批业务单据
- 复杂报表分析
- 独立商机跟进记录模型

## 9. 商机相关规则

第一版不新增独立“商机跟进记录”。

商机相关沟通内容统一沉淀到客户跟进记录中。Agent 可基于该跟进记录继续执行：

- 创建商机
- 推进商机阶段
- 标记赢单
- 标记输单

创建商机时，Agent 只追问系统必填且无法可靠推断的信息，例如商机名称、预计金额、用户数、授权模式、订阅年限、采购类型、预计成交日期等。

推进商机阶段、赢单、输单属于业务推进动作，必须先创建客户跟进记录，并在执行前获得用户确认。

## 10. 回款相关规则

用户输入类似“光大证券今天回款了”时：

1. Agent 识别客户和回款事实。
2. 客户唯一明确时，自动创建跟进记录。
3. 查询客户合同、回款计划、历史回款。
4. 判断是否已有可登记的回款计划。
5. 如果有明确回款计划，补齐金额、日期、佣金归属人等必填字段。
6. 执行回款登记前展示确认摘要。
7. 用户确认后调用系统 API 登记回款。

佣金归属人规则：

- 优先使用客户协作成员。
- 如果没有协作成员，则使用客户 owner。
- 当前协作成员只允许 1 个，因此不需要多人选择。
- 执行回款登记前，确认摘要必须展示佣金归属人。

## 11. 发票相关规则

创建发票申请需要基于客户、合同、回款计划、发票抬头、开票金额、发票类型等信息。

Agent 应：

1. 查询客户发票抬头。
2. 查询合同和回款计划。
3. 判断是否存在可申请发票的回款计划。
4. 缺少发票抬头时，可引导创建发票抬头。
5. 发票申请属于业务推进动作，需要跟进记录上下文和用户确认。
6. 创建后按现有 API 返回状态；如果后端自动进入审批，则反馈审批状态。

发票抬头创建规则：

- 客户资料维护类动作，不强制创建跟进记录。
- 信息完整时可以直接创建。
- 如果系统中已存在相同纳税人识别号，不创建新抬头，提示已存在并展示已有抬头信息。
- 第一版不做修改已有发票抬头。

## 12. License 相关规则

创建 License 申请属于业务推进动作，必须先有客户跟进记录上下文。

Agent 创建 License 申请前，需要确认部署信息：

- 如果已有部署信息，展示并让用户确认使用哪个部署。
- 如果没有部署信息，先引导创建部署信息。
- License 类型为正式版时，需要关联合同。
- License 到期时间必须晚于当前日期。
- 创建 License 申请后，因现有 API 是草稿创建，Agent 需要继续询问用户是否提交审批。
- 用户确认后，调用系统已有提交审批 API。

## 13. 客户资料维护规则

创建联系人：

- 客户必须存在且唯一。
- 必填字段缺失时追问。
- 如果同一客户下存在疑似重复联系人，先提示候选重复项，让用户确认是否仍创建。
- 第一版不修改已有联系人。

创建部署信息：

- 客户必须存在且唯一。
- 必填字段包括部署名称、服务器地址、授权人数、是否默认部署。
- 服务器地址必须符合系统 API 校验。
- 如果已有默认部署，而用户要求新建默认部署，Agent 需要提示已有默认部署，并确认是否将新部署设为默认。

设置客户成员：

- 设置客户成员始终需要用户确认。
- 如果已有协作成员，必须提示当前成员，并确认是否替换。
- 设置后会影响协作、权限和回款佣金归属。

## 14. 查询能力

第一版支持客户维度业务摘要查询。

可查询并汇总：

- 客户信息
- 联系人
- 客户成员
- 跟进记录
- 商机
- 合同
- 回款计划
- 回款记录
- 发票申请
- 发票抬头
- 部署信息
- License 申请

查询类请求不创建跟进记录，不执行写入动作。

## 15. 会话与审计

第一版需要保存 Agent 会话和执行记录，包括：

- 用户输入
- Agent 识别出的意图
- 客户匹配结果
- 候选业务对象
- Agent 建议
- 用户确认或拒绝
- 缺失字段追问
- 工具/API 调用记录
- API 执行结果
- 错误信息
- 未完成任务状态

这些记录用于审计、问题追溯、继续对话和 Agent 效果评估。业务事实仍以系统业务表为准。

## 16. 跨天继续

第一版支持跨天继续未完成任务。

继续时必须：

1. 恢复任务状态、已确认字段、候选对象和上下文。
2. 重新调用系统 API 校验客户、商机、合同、回款计划、发票、部署、License、成员等最新状态。
3. 重新校验当前用户权限。
4. 对状态变化或权限变化进行提示。
5. 不允许基于旧上下文直接执行写入。

## 17. 幂等与防重复

第一版必须具备 Agent 侧幂等和防重复能力。

要求：

- 每个写入动作保存 pending/executed/failed 状态。
- 用户重复确认时，不得重复创建。
- 网络中断或跨天恢复后，先检查是否已执行。
- 对写入类 API 不自动重试，除非后端 API 明确支持幂等键。
- 查询类和幂等校验类 API 可自动重试。
- 写入失败时，Agent 需要明确提示失败或结果不确定，并引导用户刷新业务对象确认。

## 18. 权限与 API 约束

这是第一版的强约束：

1. 所有 tool 必须调用系统后端现有 API。
2. Agent 不允许直接访问数据库。
3. Agent 不允许直接写数据库。
4. Agent 不允许绕过权限体系。
5. Agent 不允许绕过后端业务校验。
6. Agent 不允许伪造用户身份执行操作。
7. Agent 调用 API 时必须使用当前登录用户上下文。
8. API 返回权限不足、数据不存在、状态不允许时，Agent 必须如实反馈。
9. 审批、提交、创建、更新等动作全部以现有 API 行为为准。
10. 如果某个 API 创建后自动提交审批，Agent 反馈审批状态；如果创建后是草稿，Agent 再询问是否提交审批。

## 19. 第一版验收标准

1. 销售能通过统一 Agent 入口输入自然语言创建客户跟进记录。
2. 客户唯一明确时，跟进记录能自动创建。
3. 客户不唯一时，Agent 能让用户选择客户。
4. 跟进记录能识别下一步动作和下次跟进时间。
5. 跟进质量能异步评估并提示优化。
6. Agent 能基于客户上下文主动提出最多 3 个高优先级建议。
7. Agent 能完成第一版定义的业务推进动作和资料维护动作。
8. 所有写入动作均通过系统现有 API 执行。
9. Agent 不直接操作数据库，不绕过权限。
10. 会话、工具调用、确认、拒绝、失败结果可追溯。
11. 跨天继续任务时会重新校验业务状态和权限。
12. 写入动作具备 Agent 侧防重复能力。

## 20. 第一版不做事项

- 不创建客户
- 不创建合同
- 不上传合同附件
- 不上传发票文件
- 不新增独立商机跟进记录
- 不做通用编辑已有对象
- 不做删除、撤销
- 不直接审批业务单据
- 不绕过系统 API
- 不绕过权限体系
- 不直接读写数据库

这版需求的核心边界是：Agent 负责理解销售自然语言、沉淀跟进记录、推动业务动作，但所有业务执行必须回到现有 CRM API 和权限体系中完成。
