# AI 意图识别优化需求规范

> 版本：1.0 | 创建日期：2026-06-09 | 状态：需求确认

---

## 一、问题分析

### 1.1 当前架构问题

**现有方案：系统提示词硬编码规则**

```python
# 当前 ai_tool_service.py 的系统提示词
【核心规则 - 根据上下文选择工具】:
1. 如果消息开头有"[当前操作对象：客户 XXX]",说明用户正在客户页面操作
   - 用户描述沟通内容、业务进展、跟进情况 → 调用 follow_up_customer
   - 用户明确说"创建商机"、"新建商机" → 调用 create_opportunity
   ...
```

**问题：**

| 问题 | 表现 | 影响 |
|------|------|------|
| **限制灵活性** | AI 变成"规则匹配器"而非语义理解 | 无法处理非预设输入 |
| **无法穷尽** | 用户输入千变万化，硬编码覆盖不了边缘情况 | 边缘场景失效 |
| **反向约束** | 当输入偏离规则，AI 反而"不敢"选择其他工具 | 识别错误 |

### 1.2 典型失败案例

**用户输入：** `"客户通过微信确认采购，后续会有采购联系人过来走合同流程"`

**期望行为：** 创建跟进记录（这是沟通内容）

**实际行为：** 查询商机列表（AI 错误理解意图）

**根本原因：** 工具描述太简单（`"添加客户跟进记录"`），AI 无法从描述中理解"何时使用此工具"

---

## 二、成熟方案核心思路

### 2.1 设计理念

**不应该：** 用系统提示词"强制"AI 选工具

**应该：** 让工具描述本身足够清晰，AI 自行判断意图

```
成熟架构：

完整的前后文 + 充分的 tools 描述 = AI 自行判断意图
      ↓                    ↓                ↓
   环境信息           "何时用我"          语义分析
```

### 2.2 对比

| 维度 | 硬编码规则方案 | 成熟方案 |
|------|---------------|---------|
| **系统提示词** | 强制规则（如果是 A 必须调用 B） | 指导性信息（提供背景，不强制） |
| **工具描述** | 简单（"添加跟进记录"） | 充分（何时使用、触发场景、示例） |
| **AI 角色** | 被动执行规则 | 主动语义理解 |
| **边缘场景** | 规则覆盖不了就出错 | AI 有足够信息自然能理解 |

## 三、优化方案详细设计

### 3.1 优化工具描述（核心）

**目标：** 让工具描述包含"何时使用"的信息，AI 自行判断

#### 3.1.1 描述格式规范

每个工具描述应包含三部分：

```python
{
    "name": "工具名",
    "description": """{一句话概述}

适用于：{触发场景列表}
示例：{典型输入示例}

注意：{容易混淆的场景说明}（可选）
"""
}
```

#### 3.1.2 具体优化示例

**跟进工具（follow_up_customer）**

```python
# 当前
"description": "添加客户跟进记录"

# 优化后
"description": """记录与客户的沟通互动情况。

适用于：用户描述沟通内容、业务进展、下一步安排
示例：
- "微信确认了采购意向"
- "电话讨论了报价方案"
- "客户说下周签约"
- "今天拜访客户，聊了需求"

注意：如果用户要"查看/搜索"信息，请用查询工具，而非跟进。
"""
```

**查询商机工具（query_opportunities）**

```python
# 当前
"description": "查询商机列表。支持按状态、阶段、负责人、关键词组合筛选"

# 优化后
"description": """搜索查看商机列表。

适用于：用户明确要"查看/搜索/找"商机
示例：
- "看看有哪些商机"
- "查一下跟进中的商机"
- "找找这个客户的商机"
- "搜索 CRM 相关的商机"

注意：如果用户描述的是沟通内容而非查看需求，请使用跟进工具。
"""
```

**创建商机工具（create_opportunity）**

```python
# 当前
"description": "创建新商机"

# 优化后
"description": """创建新的销售商机。

适用于：用户明确要"创建/新建/添加"商机
示例：
- "创建一个商机"
- "新建商机"
- "添加一个订阅商机"
- "帮我建个商机，50人1年"

注意：商机名称格式为：{客户名}-{用户数}人{年限}年订阅
"""
```

### 3.2 优化系统提示词（辅助）

**目标：** 提供背景信息，但不强制决策

```python
def _get_system_prompt(self) -> str:
    return """你是 CRMWolf 的 AI 助手，帮助用户管理销售业务。

【上下文理解】
消息开头可能有"[当前操作对象：XXX]"，表示用户正在该实体页面操作。
这提供了环境信息，帮助你理解用户意图。

【工具选择建议】（非强制，请根据语义判断）
- 用户描述沟通/互动 → 考虑跟进类工具
- 用户要查看/搜索 → 考虑查询类工具
- 用户要创建/新建 → 考虑创建类工具
- 用户要修改状态 → 考虑更新类工具

【重要】请仔细阅读每个工具的描述，根据用户输入的语义选择最匹配的工具。
每个工具描述中都有"适用于"和"示例"说明何时使用。

今天是 """ + self._get_current_date() + """
"""
```

### 3.3 丰富上下文信息

**目标：** 给 AI 更完整的背景信息，而非只告诉"当前实体"

```python
# 当前：只有实体信息
[当前操作对象：客户 "中国科学院信息工程研究所"（ID: 10）]

# 优化：包含更多业务背景
"""
当前环境：
- 操作对象：客户 #10（中国科学院信息工程研究所）
- 客户状态：跟进中
- 关联商机：2个（CRM项目-跟进中，OA项目-赢单）
- 最近跟进：2026-06-05 电话沟通报价方案

用户输入：客户通过微信确认采购，后续会有采购联系人过来走合同流程
"""
```

### 3.4 代码层自动注入（确定性逻辑）

**目标：** 不依赖 AI 提取的确定性参数，代码自动处理

```python
# 用户在客户页面 → customer_id 自动注入
if entity_context and entity_context["entity_type"] == "customer":
    # 不需要 AI 从消息中提取 ID
    params["customer_id"] = entity_context["entity_id"]
    params["customer_name"] = get_customer_name(entity_context["entity_id"])
```

## 五、验收标准

### 5.1 功能验收

| 验收项 | 标准 |
|--------|------|
| 跟进识别准确率 | 用户描述沟通内容时，≥95% 选择跟进工具 |
| 查询识别准确率 | 用户明确要查看时，≥95% 选择查询工具 |
| 创建识别准确率 | 用户明确要创建时，≥95% 选择创建工具 |
| 边缘场景处理 | 非预设输入能正确理解意图 |

### 5.2 测试场景

| 场景 | 输入 | 期望工具 |
|------|------|----------|
| 沟通内容 | "微信确认采购意向" | follow_up_customer |
| 沟通内容 | "电话聊了报价" | follow_up_customer |
| 查看需求 | "看看有哪些商机" | query_opportunities |
| 创建需求 | "新建一个商机" | create_opportunity |
| 混合场景 | "今天签合同了" | follow_up_customer + win_opportunity |

## 六、风险评估

| 鹰险 | 影响 | 应对措施 |
|------|------|----------|
| 工具描述过长 | Token 消耗增加 | 描述控制在 150 字内 |
| AI 仍然误判 | 用户体验差 | 增加置信度检测 + 多候选机制 |
| 边缘场景未覆盖 | 识别错误 | 持续收集反馈，迭代优化 |
