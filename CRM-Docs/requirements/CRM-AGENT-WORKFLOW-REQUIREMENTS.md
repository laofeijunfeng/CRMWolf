## 一、需求背景

### 1.1 业务场景

销售人员在与客户沟通后，需要将沟通内容录入 CRM 并推进业务流程：

**典型场景**：
```
用户输入："微信跟进客户 POC 情况，客户反馈产品适用，确认采购，后续会有采购人员联系推进合同流程"
```

**期望系统处理**：
1. 创建跟进记录（沟通内容、跟进方式）
2. 检查客户商机状态
3. 展示商机列表让用户选择
4. 标记选中商机为赢单
5. 询问用户是否创建合同
6. 用户确认后创建合同草稿

### 1.2 当前痛点

| 痛点 | 当前实现 | 影响 |
|------|----------|------|
| **单轮单工具** | 只执行第一个工具，后续意图丢失 | 用户需要多次操作 |
| **无上下文预加载** | AI 不知道客户是否有商机 | AI 无法做出正确判断 |
| **实体歧义无处理** | 多商机时盲目选择第一个 | 可能操作错误商机 |
| **无暂停-等待机制** | 缺少信息时直接报错 | 用户体验差 |
| **AI 不可控** | 依赖 AI 记忆和推理 | 流程执行不稳定 |

### 1.3 核心挑战

**如何在赋予 AI 灵活性的同时，确保业务流转的确定性和安全性？**

## 二、核心原则

### 2.1 认知隔离原则

**AI 只负责"识别意图"，不负责"执行决策"**

| 职责 | 控制方 | 说明 |
|------|--------|------|
| **意图识别** | AI | 理解用户输入，输出意图 + 置信度 + 证据链 |
| **流程编排** | Workflow Engine (代码) | 决定执行顺序、校验状态流转 |
| **执行操作** | Execution Engine (代码) | 权限校验、幂等性、审计 |

**关键约束**：
- AI 严禁直接执行破坏性操作（删除、强制归档）
- AI 的所有推理结论必须附带置信度
- 低置信度（<70%）必须人工确认

### 2.2 分层交互原则

**根据风险等级差异化确认策略**

| 风险级别 | 操作类型 | 交互方式 | 示例 |
|----------|----------|----------|------|
| **LOW** | 信息查询、只读操作 | 静默执行 | 查询商机列表、获取客户详情 |
| **MEDIUM** | 信息补充、低风险更新 | Toast 通知 | 创建跟进记录、更新备注 |
| **HIGH** | 状态变更、金额修改 | Modal 强确认 | 标记赢单、创建合同、改价 |
| **CRITICAL** | 不可逆操作、删除 | Modal + 双重确认 | 删除客户、终止合同 |

### 2.3 架构防御原则

**推理与执行解耦，系统可抵御 AI 不可控性**

| 机制 | 说明 |
|------|------|
| **幂等性** | 所有操作可重复执行，不产生副作用 |
| **可回滚** | 关键节点支持一键回滚（Rollback） |
| **熔断机制** | AI 连续失败时自动切换人工接管 |
| **状态快照** | 执行前保存快照，用于回滚和审计 |

### 2.4 流程刚性原则

**AI 建议服从于业务工作流的物理约束**

| 约束类型 | 说明 | 示例 |
|----------|------|------|
| **状态机校验** | 禁止跳过中间节点 | 赢单前必须有商机存在 |
| **业务不变式** | 强制校验业务规则 | 合同金额 ≤ 商机金额 |
| **必填字段校验** | 缺少必填字段时询问用户 | 创建商机需要产品、金额 |

### 2.5 演进稳定性原则

**系统具备自我纠错和版本管理能力**

| 机制 | 说明 |
|------|------|
| **全链路可观测** | 记录 AI Thought → Workflow → Execution 完整链路 |
| **Prompt 版本化** | Prompt、Tools 定义支持版本控制、灰度发布 |
| **快速回滚** | 发现问题立即回退到稳定版本 |

```
│ - 赢单前必须有商机                                               │

│ - 回款必须关联合同                                               │
│                                                                 │

│ - Required 步骤失败 → 重试 3 次 → 回滚到上一节点                 │
│ - Optional 步骤失败 → 跳过继续                                   │

    ↓
┌─────────────────────────────────────────────────────────────────┐

│ - 从快照恢复状态                                                 │

    ↓

│ - Prompt 版本（v1.1）                                            │
│ - 置信度分布                                                     │
│ - 执行时间、失败率                                               │
│ - 用户修正行为（用于后续优化）                                   │
│                                                                 │
│ 监控机制：                                                       │
│ - 熔断：失败率 >30% → 自动切换人工接管                           │
│ - 灰度：新 Prompt 逐步上线（20% → 50% → 100%）                   │
│ - 回滚：发现问题立即回退到上一稳定版本                           │
└─────────────────────────────────────────────────────────────────┘
```

## 四、核心业务流程

### 4.1 客户确认采购（赢单场景）

**流程定义**：

| Step | 操作 | 类型 | 前置条件 | 后置处理 |
|------|------|------|----------|----------|
| 1 | create_follow_up | required | 无 | 记录沟通内容 |
| 2 | get_entity_context | required | 无 | 获取商机列表 |
| 3 | check_opportunity | required | 无 | 分支判断 |
| 4a | ask_create_opportunity | required | 无商机 | 用户确认创建商机 |
| 4b | ask_confirm_win | required | 1个商机 | 用户确认赢单 |
| 4c | ask_select_opportunity | required | 多商机 | 用户选择商机 |
| 5 | win_opportunity | required | 商机存在 | 标记赢单 |
| 6 | ask_create_contract | optional | 赢单成功 | 用户确认创建合同 |
| 7 | create_contract | optional | 用户确认 | 创建合同草稿 |

**状态机约束**：
- 赢单前必须有商机存在
- 合同必须关联已赢单商机
- 回款必须关联合同

### 4.2 商机歧义消解

**触发条件**：客户有多个跟进中商机，用户要操作某个商机

**处理流程**：

| 商机数量 | 处理方式 |
|----------|----------|
| 0 个 | 询问用户是否创建商机 |
| 1 个 | 展示商机信息，询问用户确认赢单 |
| >1 个 | 展示商机列表，让用户选择 |

**ask_user 格式**：
```json
{
    "question": "客户有多个跟进中商机，请选择确认采购的是哪个？",
    "options": [
        "CRM项目-50人1年订阅（30万）",
        "OA项目-30人1年订阅（18万）"
    ],
    "metadata": {
        "candidates": [
            {"id": 101, "name": "CRM项目...", "amount": 300000},
            {"id": 102, "name": "OA项目...", "amount": 180000}
        ]
    }
}
```

### 4.3 信息缺失处理

**触发条件**：缺少必填字段

**处理流程**：

| 缺失场景 | 询问内容 |
|----------|----------|
| 创建商机无产品名称 | "请补充产品名称" |
| 创建商机无金额 | "请补充预计金额" |
| 创建合同无金额 | "请补充合同金额" |
| 创建合同无签约日期 | "请补充签约日期" |

## 五、业务不变式定义

### 5.1 金额约束

| 不变式 | 校验规则 | 错误处理 |
|--------|----------|----------|
| 合同金额 ≤ 商机金额 | contract.amount ≤ opportunity.amount | 拦截并询问用户确认 |
| 回款金额 ≤ 合同金额 | payment.amount ≤ contract.amount | 拦截并询问用户确认 |
| 发票金额 ≤ 合同金额 | invoice.amount ≤ contract.amount | 拦截并询问用户确认 |

### 5.2 关联约束

| 不变式 | 校验规则 | 错误处理 |
|--------|----------|----------|
| 赢单必须关联商机 | win_opportunity.opportunity_id 存在 | 先询问用户创建商机 |
| 合同必须关联商机 | contract.opportunity_id 存在且状态为 WON | 先标记赢单 |
| 回款必须关联合同 | payment.contract_id 存在 | 先创建合同 |

### 5.3 状态流转约束

| 实体类型 | 合法状态流转 |
|----------|-------------|
| Lead | NEW → FOLLOWING → CONVERTED → (终态) |
| Opportunity | FOLLOWING → WON/LOST → (终态) |
| Contract | DRAFT → PENDING → SIGNED → EXECUTED → TERMINATED |

```typescript
    data: {
        session_id: string;
        step_id?: string;
        intent?: object;
        result?: object;
        question?: string;
        options?: string[];
        confidence?: number;
        message?: string;
        rollback_reason?: string;
    }
}
```

## 七、监控与运维

### 7.1 监控指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **意图识别准确率** | ≥85% | AI 正确识别用户意图的比例 |
| **流程完成率** | ≥95% | 流程完整执行不中断的比例 |
| **回滚率** | ≤3% | 因错误导致回滚的比例 |
| **熔断触发率** | ≤1% | 系统切换人工接管的比例 |
| **平均流程时长** | ≤30s | 从开始到完成的时间 |

### 7.2 熔断机制

| 触发条件 | 处理方式 |
|----------|----------|
| AI 连续失败 3 次 | 暂停当前流程，提示用户手动操作 |
| 流程执行失败率 >30%（10分钟内） | 全局熔断，切换人工模式 |
| AI 推理置信度 <50% | 强制人工确认 |

### 7.3 灰度发布策略

| 阶段 | 用户比例 | 验证内容 |
|------|----------|----------|
| 内测 | 5-10 人 | 功能完整性、边界场景 |
| 灰度 1 | 20% 用户 | 稳定性、性能 |
| 灰度 2 | 50% 用户 | 准确率、用户反馈 |
| 全量 | 100% 用户 | 生产验证 |

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **意图识别准确率** | 80-85% | 明确意图 95%+，模糊意图 70%+ |

| **系统健壮性** | 回滚率 <3% | 架构防御保证 |
| **用户干扰度** | 每 10 次建议 ≤2 次强确认 | 分层交互保证 |

## 十、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| AI 误判意图 | 流程执行错误 | 置信度校验 + 强确认机制 |
| AI 遗忘上下文 | 决策不完整 | Session State 持久化 + 每轮重新注入 |
| Workflow 死循环 | 资源耗尽 | max_steps + timeout + 熔断 |
| Prompt 版本波动 | 系统不稳定 | 版本控制 + 灰度发布 + 快速回滚 |

## 十一、实施阶段
