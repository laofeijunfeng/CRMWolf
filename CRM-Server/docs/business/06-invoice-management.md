# 发票管理模块

## 1. 模块概述

### 1.1 产品背景

发票管理是CRM系统与财务流程衔接的关键环节。本模块旨在为销售团队提供一站式的发票管理能力，实现从开票申请、财务审批到与回款计划关联的全流程数字化管理。通过标准化开票流程、关联业务上下文（客户、合同、回款），极大提升开票效率与准确性。

### 1.2 核心价值

- **流程标准化**：规范开票申请与审批流程，降低合规风险
- **数据关联化**：打通"客户-商机-合同-回款-发票"数据链条，形成完整业务闭环
- **操作灵活化**：支持"先票后款"与"先款后票"两种业务模式，适配不同客户结算习惯

## 2. 核心业务流程

### 2.1 业务流程图

```
销售成员维护客户开票抬头
    ↓
合同审批通过后
    ↓
销售提交开票申请
    ↓
选择关联回款计划
    ↓
系统自动关联客户/商机/合同
    ↓
财务审批发票申请
    ↓
发票开具并更新状态
    ↓
回款时关联已开发票
```

### 2.2 业务规则

#### 发票创建规则

1. **关联关系自动建立**
   - 通过 payment_plan_id 找到对应的回款计划
   - 通过回款计划找到对应的 contract_id
   - 通过合同找到对应的 customer_id 和 opportunity_id
   - 验证 invoice_title_id 确实属于该客户

2. **开票抬头管理**
   - 支持一个客户添加多个开票抬头
   - 同一客户下纳税人识别号不能重复
   - 支持设置默认抬头
   - 开票抬头类型：COMPANY(单位)、PERSONAL(个人)

#### 发票状态流转

```
DRAFT (草稿)
    ↓ 提交审批
PENDING_REVIEW (待审批)
    ↓ 财务审批
    ├─→ APPROVED (已批准) → 标记开票 → ISSUED (已开票)
    └─→ REJECTED (已拒绝)
```

**状态说明：**
- DRAFT：草稿状态，可编辑、可删除
- PENDING_REVIEW：待审批，已提交审批，不可编辑
- APPROVED：已批准，等待开票
- REJECTED：已拒绝，需要重新提交
- ISSUED：已开票，最终状态

#### 回款关联发票规则

在回款登记时，支持关联多个发票。校验规则：

1. 发票必须属于同一客户和合同
2. 发票总金额不能超过回款金额
3. 发票状态必须为 ISSUED

## 3. 数据模型设计

### 3.1 开票抬头表 (crm_invoice_titles)

存储客户的一个或多个开票抬头信息。

| 字段名 | 类型 | 必填 | 描述 |
|--------|------|------|------|
| id | bigint | 是 | 主键（自增） |

| title | varchar(255) | 是 | 开票抬头 |
| taxpayer_id | varchar(100) | 是 | 纳税人识别号 |
| bank_name | varchar(255) | 否 | 开户行 |
| bank_account | varchar(100) | 否 | 开户账号 |
| address | varchar(500) | 否 | 开票地址 |
| phone | varchar(50) | 否 | 电话 |
| is_default | tinyint | 是 | 是否默认抬头（0:否, 1:是） |
| created_time | datetime | 是 | 创建时间 |
| last_modified_time | datetime | 是 | 最后修改时间 |

| 字段名 | 类型 | 必填 | 描述 |
|--------|------|------|------|
| id | bigint | 是 | 主键（自增） |
| application_number | varchar(50) | 是 | 申请单号（系统生成） |

| contract_id | bigint | 是 | 关联合同ID |
| opportunity_id | bigint | 是 | 关联商机ID |
| payment_plan_id | bigint | 是 | 关联回款计划ID |
| payment_record_id | bigint | 否 | 关联回款记录ID |
| invoice_title_id | bigint | 是 | 开票抬头ID |
| invoice_amount | decimal(12,2) | 是 | 开票金额 |
| invoice_type | varchar(20) | 是 | 发票类型 |
| status | varchar(20) | 是 | 申请状态 |
| applicant_id | varchar(100) | 是 | 申请人ID（飞书用户ID） |
| reviewer_id | varchar(100) | 否 | 审批人ID（飞书用户ID） |
| review_comment | varchar(500) | 否 | 审批意见 |

| last_modified_time | datetime | 是 | 最后修改时间 |

**申请单号生成规则：** INV-yyyyMMdd-0001

## 6. 业务指标

### 6.1 关键指标

- **开票申请总数**：系统内所有发票申请的数量
- **待审批发票数**：状态为PENDING_REVIEW的发票数量
- **本月开票金额**：本月内创建的发票总金额
- **开票完成率**：已开票金额 / 计划开票金额
- **平均审批时长**：从提交到审批完成的平均时间

### 6.2 统计维度

- 按客户统计开票金额和发票数量
- 按合同统计开票进度
- 按时间统计开票趋势（日、周、月）
- 按发票类型统计分布（专用发票 vs 普通发票）

## 7. 常见问题

### Q1: 发票申请提交后可以修改吗？

A: 只有草稿（DRAFT）状态的发票申请可以修改和删除。一旦提交审批（PENDING_REVIEW）或后续状态，将无法修改。如需修改，需拒绝后重新创建。

### Q2: 一个回款计划可以关联多个发票吗？

A: 可以。一个回款计划可以关联多个发票申请，但所有发票的总金额不能超过回款计划金额。

### Q3: 发票申请被拒绝后如何处理？

A: 被拒绝（REJECTED）的发票申请无法直接重新提交。需要查看拒绝原因，修改相关信息后创建新的申请。

### Q4: 如何处理"先票后款"和"先款后票"两种场景？

A: 
- **先票后款**：先创建发票申请，审批开票后再进行回款登记
- **先款后票**：先进行回款登记，回款时可以关联已有发票，或后续补开发票申请

### Q5: 开票抬头可以设置多个吗？

A: 可以。一个客户可以添加多个开票抬头（如不同分公司、不同项目等），并可以设置其中一个为默认抬头。

## 8. 最佳实践

### 8.1 发票申请管理

1. **统一维护开票抬头**：建议在客户首次合作时，完整维护开票抬头信息，避免每次开票时临时录入
2. **设置默认抬头**：为常用客户设置默认抬头，提升开票申请效率
3. **定期核对信息**：定期检查开票抬头信息的准确性，特别是纳税人识别号
4. **关联回款计划**：创建发票申请时务必正确选择关联回款计划，确保财务数据一致性

### 8.2 审批流程优化

1. **明确审批标准**：财务团队应制定明确的发票审批标准和SOP
2. **及时处理审批**：建议财务团队每日定时处理待审批发票，避免业务延误
3. **记录审批意见**：无论是批准还是拒绝，都应填写明确的审批意见，便于后续追溯

### 8.3 数据分析

1. **监控开票进度**：定期查看各合同的开票进度，确保与业务进展同步
2. **分析开票类型**：分析专用发票和普通发票的使用情况，优化开票策略
3. **评估审批效率**：监控平均审批时长，持续优化审批流程
