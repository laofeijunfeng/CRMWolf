## 三、关键决策点分析

### 3.1 用户决策点 1：License 类型选择

**场景**: 用户在申请表单中选择"试用"或"正式" License

**决策触发**:
- 用户点击 License 类型单选按钮
- 表单字段动态切换（合同字段显示/隐藏）

**决策影响**:
- 试用 License → 合同字段隐藏，流程简化
- 正式 License → 合同字段显示，需校验合同状态

**UX 设计要求**（参照 UI/UX Pro Max §8 Forms & Feedback）:

1. **渐进披露（Progressive Disclosure）**:
   ```
   ┌─────────────────────────────────────────────────────┐
   │ License 类型                                        │
   │ ○ 试用 License (TRIAL)                              │
   │   └─────────────────────────────────────           │
   │   说明: 适用于 POC 试用场景，无需关联合同          │
   │                                                     │
   │ ● 正式 License (OFFICIAL)                          │
   │   └─────────────────────────────────────           │
   │   说明: 适用于正式部署，必须关联合同               │
   │                                                     │
   │   ⚠️ 关联合同（新增字段）                           │
   │   [下拉选择] HT-2026-001 销售合同                  │
   │                                                     │
   └─────────────────────────────────────────────────────┘
   ```

2. **即时反馈**:
   - 切换类型时，合同字段立即显示/隐藏（150ms 过渡动画）
   - 切换为试用时，清空已选合同（提示："已清空合同选择"）

3. **上下文提示**:
   - 正式 License 无合同时，显示空状态 + 操作建议
   - 试用 License 显示简化说明，降低认知负担

---

### 3.2 用户决策点 2：审批通过/拒绝

**场景**: 审批人在审批详情页点击"通过"或"拒绝"

**决策触发**:
- License 信息填写完整（通过按钮启用）
- License 信息为空（通过按钮禁用）

**决策影响**:
- 通过 → 状态变为 ISSUED，客户到期时间更新，申请人收到通知
- 拒绝 → 状态变为 REJECTED，申请人需重新申请

**UX 设计要求**（参照 UI/UX Pro Max §2 Touch & Interaction）:

1. **破坏性操作分离（Destructive Emphasis）**:
   ```
   ┌─────────────────────────────────────────────────────┐
   │                                                     │
   │              [拒绝]  [通过并发放授权码]              │
   │              ↑        ↑                             │
   │              │        │                             │
   │           红色按钮   绿色按钮                        │
   │           左侧分离   右侧强调                        │
   │                                                     │
   └─────────────────────────────────────────────────────┘
   ```

2. **确认对话框**:
   - 拒绝操作需二次确认（防止误操作）
   - 确认文案："确认拒绝该 License 申请？申请人需重新申请。"

3. **禁用状态明确**:
   - License 信息为空时，通过按钮禁用 + 提示："请填写完整授权码信息"
   - 禁用按钮使用 `opacity: 0.5` + `cursor: not-allowed`

## 四、错误恢复路径设计

### 4.1 申请提交失败

**场景**: 申请提交时校验失败或 API 异常

**错误类型**:
| 错误类型 | 原因 | 恢复路径 |
|----------|------|----------|
| 合同校验失败 | 合同状态非生效 | 跳转合同详情 + 提示"需创建生效合同" |
| 到期时间校验失败 | 到期时间 ≤ 今天 | 自动聚焦日期选择器 + 错误提示 |
| 部署信息缺失 | 部署信息已删除 | 弹窗提示"部署信息已删除，请重新配置" |
| 网络异常 | API 请求失败 | Toast 提示"提交失败，请稍后重试" + 保留表单数据 |

**UX 设计要求**（参照 UI/UX Pro Max §8 Forms & Feedback）:

1. **错误定位（Error Placement）**:
   - 错误提示显示在相关字段下方（而非全局顶部）
   - 使用红色文字 + 图标（`CircleCloseFilled`）

2. **自动聚焦（Focus Management）**:
   - 提交失败后，自动聚焦第一个错误字段
   - 错误字段使用红色边框 + shake 动画（100ms）

3. **表单自动保存（Form Autosave）**:
   - 提交失败后，表单数据保留（不刷新页面）
   - 提示："表单数据已保存，可修改后重新提交"

---

### 4.2 审批通过后授权码格式错误

**场景**: 审批人填写的 License 信息格式不正确，解析失败

**错误处理**:
```
┌─────────────────────────────────────────────────────┐
│ ⚠️ License 信息格式校验失败                         │
├─────────────────────────────────────────────────────┤
│                                                     │
│ 缺失字段:                                           │
│ - 企业编号（必填）                                  │
│ - 支持模块（必填）                                  │
│ - 服务端 License（必填）                            │
│ - 客户端 License（必填）                            │
│                                                     │
│ 操作建议:                                           │
│ [查看格式模板]  [返回修改]                          │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**恢复路径**:
1. 显示格式校验错误详情（缺失字段列表）
2. 提供"查看格式模板"按钮（打开示例模板）
3. 提供"返回修改"按钮（返回 License 信息填写）
4. License 信息保留（不清空）

---

### 4.3 Word 导出失败

**场景**: License 文档导出失败（部署信息缺失、文件损坏等）

**错误处理**:

1. 显示错误原因（部署信息缺失）
2. 提供手动补充信息选项（弹窗填写缺失字段）
3. 提供重新配置部署选项（跳转部署信息管理）
4. 提供联系客服选项（客服链接）

## 五、心理模型设计

### 5.1 销售人员心理模型

**用户画像**:
- 角色: 销售成员/销售总监
- 目标: 快速为客户申请 License，完成部署
- 焦虑点: 审批进度不可见，申请流程繁琐

**心理模型**:
```
┌─────────────────────────────────────────────────────┐
│ 销售人员心理模型                                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│ 预期:                                               │
│ - 配置部署信息是一次性工作（类似发票抬头）          │
│ - 申请 License 是简单流程（选部署 → 填信息 → 提交） │
│ - 审批进度应该实时可见（类似合同审批）              │
│ - 获取授权码后应该能快速导出文档                    │
│                                                     │
│ 担忧:                                               │
│ - 没有合同能否申请 License？                        │
│ - 审批被拒绝后如何重新申请？                        │
│ - 部署信息删除后申请会怎样？                        │
│ - 导出文档格式是否正确？                            │
│                                                     │
│ 期望反馈:                                           │
│ - 提交后立即知道审批人是谁                          │
│ - 审批进度实时更新（飞书通知 + 页面刷新）           │
│ - 拒绝时提供明确原因和修改建议                      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**UX 设计响应**:
1. **降低认知负担**:
   - 部署信息配置前置（在申请前完成）
   - 申请表单简化（试用 License 无合同字段）

2. **提供进度可见性**:
   - 申请列表显示审批进度（状态徽章 + 审批人头像）
   - 飞书通知实时提醒（审批通过/拒绝）

3. **提供撤销/重试能力**:
   - 草稿状态可删除/重新提交
   - 拒绝后提供修改建议 + 快捷重申请入口

---

### 5.2 审批人心理模型

**用户画像**:
- 角色: 团队所有者/销售总监/财务
- 目标: 快速审批 License 申请，填写授权码
- 焦虑点: 授权码格式复杂，粘贴错误率高

**心理模型**:
```
┌─────────────────────────────────────────────────────┐
│ 审批人心理模型                                       │
├─────────────────────────────────────────────────────┤
│                                                     │
│ 预期:                                               │
│ - 审批流程简单（查看申请 → 填写授权码 → 通过）      │
│ - License 信息格式固定（企业编号 + 授权码）         │
│ - 申请人已填写必要信息（客户、部署、合同）          │
│                                                     │
│ 担忧:                                               │
│ - License 信息格式是否正确？                        │
│ - 填写错误后能否修改？                              │
│ - 审批通过后申请人能否收到完整信息？                │
│                                                     │
│ 期望反馈:                                           │
│ - License 信息填写框有格式提示                      │
│ - 提交前有格式校验（防止错误）                      │
│ - 通过后有确认反馈（申请人已收到通知）              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**UX 设计响应**:
1. **提供格式模板**:
   - License 信息输入框右侧显示格式示例
   - 提供"复制模板"按钮（一键复制格式）

2. **提供实时校验**:
   - 提交前校验 License 信息格式
   - 缺失字段实时提示（企业编号、授权码）

3. **提供确认反馈**:
   - 审批通过后显示"申请人已收到通知"

## 八、撤销/重试能力设计

### 8.1 草稿状态撤销

**场景**: 用户创建申请但未提交，想要删除草稿

**操作路径**:
```
申请列表 → 草稿申请卡片 → [删除] 按钮 → 确认对话框 → 删除成功
```

**UX 要求**:
- 删除前二次确认（防止误操作）
- 确认文案："确认删除该草稿申请？删除后数据无法恢复。"
- 删除成功 Toast："草稿申请已删除"

---

### 8.2 拒绝后重新申请

**场景**: 申请被拒绝，用户想要修改后重新申请

**操作路径**:
```
申请列表 → 已拒绝申请卡片 → [查看原因] → [重新申请] 按钮 →
复制原申请数据 → 修改字段 → 重新提交
```

**UX 要求**:
- 拒绝卡片显示拒绝原因（审批意见）
- "重新申请"按钮一键复制原申请数据（部署信息、备注等）
- 自动打开申请表单（填充原数据）
- 用户修改问题字段后重新提交

---

### 8.3 审批中撤回

**场景**: 申请已提交但审批人未处理，用户想要撤回修改

**操作路径**:
```
申请详情 → [撤回申请] 按钮 → 确认对话框 →
状态更新: PENDING → DRAFT → 修改申请 → 重新提交
```

**UX 要求**:
- 仅 PENDING 状态允许撤回（审批人未处理）
- 撤回前二次确认："确认撤回申请？撤回后需重新提交审批。"
- 撤回成功 Toast："申请已撤回，可修改后重新提交"
