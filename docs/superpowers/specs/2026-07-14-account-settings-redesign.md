# 账户设置页面重构设计文档

**版本**: V1.0  
**创建日期**: 2026-07-14  
**状态**: 设计已确认，待实施计划评审

---

## 一、背景与目标

现有侧栏左下角用户菜单将“个人资料”和“账户设置”作为两个入口，旧 `Settings.vue` 又将个人账户、系统管理入口、操作记录和退出登录混合在同一个 Element Plus 页面中，信息架构不清晰，也不符合当前 shadcn-vue 与 V2 设计系统迁移方向。

本次重构以独立的新页面承载个人账户能力：

- 将用户菜单中的“个人资料”和“账户设置”收拢为唯一的“账户设置”入口。
- 新建账户设置页面，展示个人信息、账户详情，并提供修改密码。
- 将退出登录保留在侧栏左下角用户菜单，使用确认弹窗后完成本地状态清理与跳转。
- 采用 `variables-v2.scss` Design Tokens 和已安装的 shadcn-vue 组件；不基于旧 `Settings.vue` 重构或改造。

## 二、范围

### 2.1 本次包含

1. 新建受认证保护的账户设置路由和页面。
2. 调整侧栏底部用户菜单：保留团队切换，个人设置区只保留“账户设置”和“退出登录”。
3. 在新页面中展示个人信息、账户详情和安全设置。
4. 使用现有 `authApi.changePassword` 对接修改密码。
5. 在用户菜单中接入退出登录确认、状态清理和跳转登录页。
6. 对直接访问或刷新账户设置页提供用户信息加载、失败与重试状态。

### 2.2 明确不包含

- 不迁移、不修改旧 `Settings.vue` 中的角色管理、审批流程、采购配置、AI 配置、通知配置、团队成员等管理能力。
- 不在账户设置页面提供个人资料编辑；当前没有自助资料更新 API。
- 不实现服务端令牌吊销；当前后端没有 logout/revocation 端点。
- 不迁移旧页面的“我的操作记录”。
- 不新增 Element Plus 组件或依赖。

## 三、信息架构与路由

### 3.1 用户旅程

```
桌面端侧栏左下角用户信息
  └─ hover / 键盘操作打开向上菜单
       ├─ 团队切换（保留现有功能）
       ├─ 账户设置 → /settings/account
       └─ 退出登录 → confirmLogout() 确认 → 清理状态 → replace(/login)

移动端可见的个人/更多入口
  └─ 账户设置 → /settings/account
  └─ 退出登录 → confirmLogout() 确认 → 清理状态 → replace(/login)
```

### 3.2 路由

新增认证路由：

```ts
{
  path: 'settings/account',
  name: 'AccountSettings',
  component: () => import('@/views/AccountSettings.vue'),
  meta: { requiresAuth: true, title: '账户设置' },
}
```

该路由嵌套在既有 `AppLayout` 下，继续使用既有路由守卫。用户菜单的唯一账户入口导航到该路由；不再导航到 `/settings/profile`。

页面内容区保留唯一语义 `<h1>`“账户设置”。`AppLayout` 的 TopBar 若同时渲染路由标题，只作为视觉上下文而不产生第二个语义标题，或在该路由隐藏其中心标题。路由完成后，应将键盘焦点移到主内容区或该 `<h1>`，让读屏用户获知页面已切换。

## 四、侧栏用户菜单

### 4.1 入口行为

账户菜单必须基于已安装的 shadcn-vue `DropdownMenu` 及其 Trigger / Content / Item 原语实现，不能继续以自定义菜单容器、事件或焦点逻辑替代组件原生语义。触发器使用原生 `button` 或组件提供的 trigger，不使用仅带 `role="button"` 的 `div`。

`AppLayout` 中的用户信息区域继续显示头像、姓名、当前团队和展开指示。桌面端允许以对 `DropdownMenu` 的有限事件适配实现悬停打开，但不得重写其焦点管理、键盘导航、Esc 关闭或原生动效；触发器通过 Tab 可获得焦点，Enter 或 Space 可打开。菜单向上展开，关闭后焦点回到触发器，菜单项按视觉顺序可键盘操作并具有可见 focus 状态。

团队切换区保持当前实现与行为不变。

移动端侧栏隐藏时，必须在现有可见的个人或“更多”入口中提供同等的“账户设置”和“退出登录”动作；不得以不可见的侧栏菜单作为移动端唯一入口。实施前应确认 `BottomNav` 的承载位置，并优先复用其现有个人/更多导航模式。

### 4.2 个人设置区

个人设置区固定为：

1. **账户设置**：使用 `Settings` 图标，关闭菜单并路由跳转至 `/settings/account`。
2. **退出登录**：使用 `LogOut` 图标和 destructive 语义，点击后关闭菜单并打开 `AlertDialog`。

删除“个人资料”菜单项及其处理函数。账户设置菜单项必须是唯一的个人账户入口。

### 4.3 退出登录

退出确认必须调用设计系统规定的 `confirmLogout()`；该统一工具负责一致的确认文案、取消/确认语义与 destructive 样式，不在 `AppLayout` 内另行实现一套 `AlertDialog`。确认内容明确说明退出后需要重新登录。

- 取消：关闭确认弹窗，不变更客户端状态。
- 确认：调用统一的登出动作，清理用户 token 和用户信息、权限缓存、团队缓存；随后使用 `router.replace('/login')` 并提示“已退出登录”。
- 路由跳转若失败，客户端状态仍应保持清理状态，并使用 `window.location.assign('/login')` 兜底，避免已退出用户通过浏览器后退返回受保护页面。

当前后端不存在 token 吊销 API，因此本次不发起网络退出请求。清理操作应集中为可复用的登出动作；所需的 team / permissions 状态清理必须通过对应 store 中类型完整的 action 完成，不得由 UI 直接改写 store state。

## 五、新账户设置页面

### 5.1 页面结构

新建 `AccountSettings.vue`，不导入或复用旧 `Settings.vue` 的页面结构、Element Plus 表单、弹窗或样式。页面位于应用主内容区，使用页面自身 `24px` V2 token 间距，并在中等宽度内容列中纵向组织：

```
AccountSettings
├─ 页面标题区
│  ├─ 账户设置
│  └─ 管理您的个人资料与账户安全
├─ Card：个人信息
│  ├─ Avatar（头像 URL 或姓名首字符降级）
│  ├─ 姓名、邮箱
│  └─ 手机号、员工编号、地区、角色
├─ Card：账户详情
│  └─ 账户状态、用户 ID、创建时间、最近更新时间
└─ Card：安全设置
   ├─ 密码说明
   └─ 修改密码 Button → Dialog
```

页面保持低噪、内容优先的视觉层级：标题区仅提供上下文；卡片用于划分信息组；没有额外的装饰性图形或渐变。桌面端信息字段使用响应式定义列表网格，窄屏单列堆叠且不产生主体横向滚动。

### 5.2 个人信息卡片

个人信息卡片展示：

| 字段 | 数据来源 | 空值呈现 |
|---|---|---|
| 头像 | `userInfo.avatar_url` | 姓名首字符，未知时 `U` |
| 姓名 | `userInfo.name` | 未设置 |
| 邮箱 | `userInfo.email` | 未设置 |
| 手机号 | `userInfo.mobile` | 未设置 |
| 员工编号 | `userInfo.employee_no` | 未设置 |
| 地区 | `userInfo.region` | 未设置 |
| 角色 | `userInfo.roles` | 未设置 |

角色使用 `userStore.fetchUserInfo()` 已合并的角色结果，避免页面额外请求角色接口。多个角色以清晰的文本或现有 `Badge` 原语呈现。

### 5.3 账户详情卡片

账户详情展示账号状态、用户 ID、创建时间和最近更新时间。时间字段必须使用 API `UserResponse` 中真实存在的 `created_at` 与 `updated_at`，通过 `Intl.DateTimeFormat` 按系统 locale 格式化；空值或无效时间统一显示“未设置”。不得使用旧页面错误的 `created_time` 字段。

状态以现有 `Badge` 组件表达语义，但不在组件外硬编码颜色，并始终保留状态文字，不能只以颜色传达状态。用户 ID 使用等宽数字样式并支持复制（以已有组件或平台原生能力实现，不新增自定义 UI 原语）。

字段值较长时优先换行；若信息密度必须省略，使用省略号并提供 Tooltip 或同等方式查看完整内容，避免文字撑破卡片。头像图片加载失败时与缺失 URL 一样回退到姓名首字符；有姓名时替代文本为“{姓名}的头像”。

### 5.4 安全设置与修改密码

安全设置卡片只提供修改密码入口，不包含退出登录。

点击“修改密码”使用 shadcn-vue `Dialog` 打开表单：

| 字段 | 规则 |
|---|---|
| 当前密码 | 必填；`autocomplete="current-password"` |
| 新密码 | 必填，6–50 个字符；`autocomplete="new-password"` |
| 确认新密码 | 必填，必须与新密码一致；`autocomplete="new-password"` |

每个字段均使用关联的可见 `Label`，并提供可访问的显示/隐藏密码按钮。新密码字段持续展示“密码长度为 6–50 个字符”辅助文本。输入在失焦或提交时校验，不在每次键入时提前展示错误；提交出现多个字段错误时，焦点自动移至第一个无效字段。字段错误紧邻对应输入框，使用 `role="alert"` 或 `aria-live` 宣告给读屏工具。

提交时调用：

```ts
authApi.changePassword({
  old_password: oldPassword,
  new_password: newPassword,
})
```

提交中禁用取消外的重复提交操作，并显示组件原生 loading 反馈。成功后清空全部密码字段、关闭 Dialog 并显示成功 toast；关闭 Dialog 也应清空密码字段。任一密码字段已填写而用户尝试关闭或取消 Dialog 时，使用统一确认机制询问“放弃本次修改？”，以防误丢输入。

失败时不记录、不输出密码值，保留表单输入并显示安全、可理解的失败消息与下一步恢复建议；网络超时或结果不明确时不自动重试，提示用户重新确认账户状态或重新登录后再试，避免重复修改造成困惑。字段校验错误紧邻对应字段呈现。

## 六、数据加载与状态处理

### 6.1 正常加载

页面从 `userStore` 获取当前用户资料。若资料已存在，立即显示；若用户直接打开页面或刷新导致资料为空，调用 `userStore.fetchUserInfo()`。

### 6.2 加载、失败与重试

- 正在获取当前用户资料时，使用既有 shadcn-vue `Skeleton` 显示页面内容骨架，避免空白卡片或虚假信息。
- 获取失败时，显示 `Alert` 错误状态和“重试”按钮；重试只重新获取用户资料。
- 字段缺失不是加载失败：统一使用“未设置”，避免 `-`、空字符串等多种空态。

## 七、组件与样式约束

### 7.1 必须使用的现有原语

- 页面信息分组：`Card`。
- 用户头像：`Avatar`。
- 信息加载：`Skeleton`。
- 状态语义：`Badge`。
- 密码字段：`Label` 与 `Input`。
- 修改密码：`Dialog`。
- 登出确认：设计系统统一的 `confirmLogout()`。
- 菜单：`DropdownMenu` 及其既有原语。
- 菜单分隔：`Separator` 或既有 dropdown 原语。
- 操作按钮：`Button`。
- 反馈：项目既有 `vue-sonner` `toast`。

不新建重复的 Button、Dialog、Dropdown、Card 或其他 shadcn-vue 已提供的 UI 组件。

### 7.2 设计系统约束

- 新增样式必须导入 `@use '@/styles/variables-v2.scss' as *;`。
- 所有颜色、间距、圆角、阴影和文字样式使用 `-v2` token；不使用硬编码颜色、间距或圆角。
- 使用 `lucide-vue-next` 图标，不使用 emoji 或 Element Plus 图标。
- 保持 shadcn-vue 原生动态效果，不额外修改其过渡时长、曲线或动画。
- 交互控件提供可见 focus 状态，图标按钮和头像提供可访问名称；小屏点击目标不小于 44×44pt。
- 亮色与暗色 token 下的正文、Badge、分隔线和 focus ring 都必须独立验证对比度；动效必须尊重 `prefers-reduced-motion`，并不应阻碍操作。

## 八、实现边界与变更清单

### 8.1 新增

- `CRM-Client/src/views/AccountSettings.vue`
- 与项目测试结构一致的账户设置页面/组件测试文件（具体路径按现有测试组织确定）。

### 8.2 修改

- `CRM-Client/src/router/index.ts`：注册 `/settings/account`。
- `CRM-Client/src/AppLayout.vue`：移除“个人资料”入口，保留唯一“账户设置”入口；以 `AlertDialog` 承载退出确认。
- `CRM-Client/src/stores/user.ts` 或专门的登出协调逻辑：保证退出时统一清理用户、权限和团队状态，且类型声明符合 store 规范。
- 可能涉及 `team`、`permissions` store 的已有清理方法，仅为完整登出状态清理服务。

### 8.3 不修改

- `CRM-Client/src/views/Settings.vue` 的旧账户、操作记录或系统管理结构。
- 任何系统管理功能与对应业务路由。
- 后端认证 API 与数据库模型。

## 九、测试与验收标准

### 9.1 自动化测试

1. 账户设置页面在已有用户资料时正确展示个人信息、账户详情和角色；缺失或无效日期统一展示“未设置”，长字段不撑破卡片，头像 URL 失败时回退到姓名首字符。
2. 用户资料为空时触发加载；加载失败展示可重试错误态，重试可恢复展示。
3. 修改密码表单阻止空值、少于 6 字符、超过 50 字符与两次密码不一致的提交；字段有可访问 Label、autocomplete、显示/隐藏按钮、字段旁 `role="alert"` 错误和首个错误聚焦。
4. 修改密码请求成功后清空表单、关闭弹窗、显示成功提示；失败时保留输入且显示包含恢复建议的错误提示，不泄露密码；关闭含输入的 Dialog 会先确认放弃修改。
5. 用户菜单中不存在“个人资料”，存在唯一“账户设置”入口，并导航至新路由；触发器可通过 Tab、Enter、Space、Esc 完成可预测操作并管理焦点。
6. 退出登录取消时不清理状态；确认时清理 user、permissions、team 状态，并以 `router.replace('/login')` 跳转登录页。

### 9.2 手动交互验收

- 桌面端 hover 和键盘焦点均能访问基于 `DropdownMenu` 的侧栏底部菜单；菜单向上展开，Esc 可关闭，焦点样式可见且关闭后归还触发器。
- 移动端存在可见且可操作的账户设置、退出登录入口；不可将隐藏的侧栏作为移动端唯一入口。
- 账户设置页面不显示旧 Settings 的系统管理入口、操作记录或退出登录按钮。
- 在 375px、768px、1024px、1440px 与小尺寸横屏下，页面保持单列/响应式可读，无页面主体横向滚动；可点击控件符合触控尺寸要求。
- 在启用 `prefers-reduced-motion`、亮色主题与暗色主题时，菜单、Dialog、文本、Badge、分隔线和 focus ring 均可辨识、可操作且符合对比度要求。
- 所有新增界面仅使用现有 shadcn-vue 原语、V2 token 和 Lucide 图标，不新增 Element Plus UI。

### 9.3 工程验证

实施完成后应依次运行：

```bash
cd CRM-Client
npm run lint
npm run type-check
npm run test:unit
```

并启动前端实际验证：从侧栏账户菜单进入新页面、成功和失败修改密码路径、取消与确认退出登录路径。验证应报告实际命令和浏览器结果；若现有未相关失败阻塞，应明确区分。
