# 设计系统迁移控制面设计

**目标：** 以 `CRM-Docs/design-system/README.md` 为唯一设计规则入口，在不破坏仍依赖 V1 的存量页面前提下，自动盘点、持续减少并最终清退 V1 令牌、旧主题、旧框架和旧规范引用。

## 权威边界

- `CRM-Docs/design-system/` 是唯一的设计规则单一事实来源（SSOT）。
- `CRM-Client/src/styles/variables-v2.scss`、`tailwind.config.ts` 和 `base.css` 是规则的运行时实现，不另行定义设计决策。
- 自动清单只陈述当前代码事实，不定义设计规则；禁止人工编辑。
- 客户端旧说明、历史计划、原型和 V1 文件均不是规则来源；在迁移期间仅可作为输入或参考，并明确标记非权威。
- V1 兼容仅为过渡状态，不新增、不扩展，台账归零后必须删除。

## 文件与脚本

```text
CRM-Docs/design-system/migration/
├── overview.md
├── compatibility.md
├── implementation-status.md
├── migration-playbook.md       # 迁移顺序、验收、清退条件
└── v1-usage-inventory.md       # 自动生成，禁止人工编辑

scripts/
├── generate-design-system-inventory.js
└── check-design-system-migration.js
```

`migration-playbook.md` 是唯一迁移操作说明。`v1-usage-inventory.md` 每次扫描完全覆盖，开头标明生成命令、生成时间、禁止手改和当前遗留总数。

## 扫描范围与分类

扫描器遍历受版本控制的 `CRM-Client/` 源码、样式、配置和文档，跳过 `node_modules`、构建产物、虚拟环境、`.git` 与临时 worktree。识别以下遗留信号：

1. V1 样式导入：`variables.scss`、`wolf-design.scss`、`element-plus-theme.scss`。
2. V1 SCSS 令牌：`$wolf-*` 且没有 `-v2` 后缀。
3. 旧框架：Element Plus 组件、图标和 API 导入。
4. 已删除设计系统路径：`MASTER.md`、`ELEMENT-PLUS-MIGRATION.md`、`design-system/pages/*.md`。
5. 旧视觉原型和非权威设计说明。

每条结果记录文件、行号、信号类型、命中内容、所属分类和建议替代方案。分类固定为：全局样式、共享组件、页面、UI 基元、测试、配置、文档/计划、原型。

输出先列出全局样式与共享组件，再按页面族汇总，最后列出非运行时引用。每个页面族包含直接信号数、依赖的共享组件和迁移验收条件。

## 分批迁移与清退

| 阶段 | 范围 | 退出条件 |
|---|---|---|
| P0 | 运行时令牌、全局样式、主题配置 | V2 映射完整，所有存量 V1 使用已进入清单 |
| P1 | 共享组件与导航 | 共享组件不再新增或保留 V1 依赖 |
| P2 | 列表页族 | 列表、筛选、表格和空状态无 V1 信号 |
| P3 | 详情页与 Sheet/Drawer | 详情布局、标签页、面板和弹层无 V1 信号 |
| P4 | 表单、审批、财务与特殊流程 | 相关页面和特有组件无 V1/旧框架信号 |
| P5 | 最终清退 | 清单归零，删除 V1 文件、旧主题、兼容说明和旧原型 |

每批迁移必须：重新生成清单、确认对应项减少、运行受影响页面测试和设计系统检查、确认没有新增遗留信号。页面迁移只在其共享依赖已迁移或有明确迁移计划时开始。

## 双阶段门禁

迁移期间，检查器比较待提交 diff 和基线：

- 修改或新增的业务源文件不得新增 V1 导入、V1 令牌、旧框架导入或旧路径引用。
- 已存在的存量信号允许保留，但必须能在自动清单中找到。
- 删除遗留信号永远允许。
- 生成清单必须与当前扫描结果一致，防止台账过期。
- 设计规则 Markdown 仍遵守现有 100 行、链接、索引和废弃路径检查。

P5 开始后切换为最终模式：任何遗留信号均失败；删除 `variables.scss`、`wolf-design.scss`、`element-plus-theme.scss`、V1 原型和兼容文件后，扫描结果必须为零。

## 原型与客户端说明

- 所有根目录导航原型和客户详情原型均非权威，P0 直接删除、不归档；需要时仅通过 Git 历史恢复。
- 被删除原型不得作为迁移依据，也不得在设计系统或代码注释中重新引用。
- `LAYOUT.md`、`ANIMATION-GUIDELINES.md`、`COMPONENTS.md`、`DETAIL-SHEET-COMPONENT.md`、客户端 `MIGRATION.md` 的规则内容需迁入对应设计系统主题；原文件随后只保留实现/API 说明或删除。
- 所有代码注释、Stylelint 说明和配置引用必须指向精确的新主题文件，不能引用已删除的章节或泛泛的历史名称。

## 验收标准

- 自动清单可重复生成，结果含完整分类、行号、替代建议与页面族汇总。
- 迁移期间无法在新修改中引入 V1/旧框架/旧路径债务。
- 迁移进度可由清单总数和每批退出条件准确判断。
- 设计系统 README 始终是唯一规则入口；外部文档和原型没有独立规则权威。
- 最终模式下扫描结果为零，V1 令牌、旧主题、兼容说明和旧原型均已删除。
- 生成器、迁移检查器、现有设计文档检查器及其测试均可在本地和 CI 运行。
